밀프레드 핵심 엔진 설계

코칭 엔진 — 자녀 데이터가 쌓일수록
정확해지는 회귀적 개인화 루프

식단 입력 → 영양·시계열 분석 → 새벽 코칭(편지·질문) → 부모 답변 → 데이터 축적 → 더 정확한 다음 코칭.
매일 도는 이 고리가 아이마다의 데이터 해자(moat)를 만든다.
개발 계획표

3단계 로드맵

이론·코칭 방법의 본진은 편식극복키트 폴더. 앱은 그 근거를 구현한다.

1

엔진 스펙 정립 (근거 수집)

  • 편식 이론 정독: SOS(Toomey)·Satter DOR·국제가이드·KDRI 2025
  • 코칭 원칙·질문 설계·프롬프트 규격을 문서화
저장: 편식극복키트 / 06_운영 / 카톡코칭
2

엔진 구현

  • 편지·질문 프롬프트 재설계 — 데이터로 아는 건 묻지 않기, 환각 차단
  • 식단 입력에 먹는 장소(집 / 어린이집·유치원) 필드 추가
  • 시계열 분석 병행 + 새벽 2~5시 크론 생성 → 아침에 도착
3

명문화

  • 코칭 엔진 개념을 용어사전에 등재
  • 이 설계 문서를 docs 허브에 게시·갱신

데이터 해자 (moat)

아이마다 쌓이는, 못 따라오는 데이터

경쟁자가 카피할 수 없는 건 기능이 아니라 그 아이의 누적 데이터다.

무엇을 먹(고 안 먹)나

식단 입력 누적 — 식재료별 노출·완식 횟수, 거부 이력, 장소별 패턴

어떻게 먹나

부모 교정(이 집 짜파게티엔 당근 빼기)·정성 답변(섞어야 먹음·영상 보면 잘 먹음)

시간에 따라

며칠째 안 먹는지·새 식재료 수용 속도·계절별 변화의 시계열

데이터가 쌓일수록 → 코칭이 그 아이에게 정확해지고 → 부모가 더 기록하고 → 데이터가 더 쌓인다 (복리)

생성 원칙

좋은 코칭 vs 나쁜 코칭

✓ 데이터로 못 푸는 것만 묻는다
실제 로그한 음식을 짚고 정성 정보를 묻는다.
"명태 드셨던데 다 드셨나요? 다른 음식과 섞어 드렸나요?"
✗ 데이터로 아는 걸 묻는다
먹었는지 여부는 식단 로그로 이미 안다.
"흰살생선 드셨나요?" (로그 보면 됨)
✓ 부모가 바꿀 수 있는 곳에 코칭
집 아침·저녁(빈약 지점)과 기관에서의 거부에 포커스.
✗ 못 바꾸는 걸 다그친다
기관 급식 메뉴는 엄마가 못 바꾼다.
✓ 정량은 전부 집계
영양평가는 어린이집·유치원 급식까지 모두 카운트.
✗ 없는 과거를 지어낸다
지난 권장 이력이 없으면 "지난번 권한 ○○"라 말하지 않는다.
