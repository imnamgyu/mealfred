1. 왜 주간 계획인가 — '매일 재계산'의 한계

현재 cron/coach는 매일 새벽 2시(KST) planFor()가 최근 7일 윈도우를 처음부터 새로 보고 그날의 시나리오·타깃·무브를 즉석에서 뽑는다. plan 엔진이 '하루 안에서의 중복'은 막았지만(시그니처 dedup·채근 방지), '한 주에 걸친 일관성'을 잡는 시간축이 없다.

현재 — 매일 독립 재계산

  • 월: 콩류 · 화: 생선 · 수: 과일 · 목: 콩류… 타깃이 휙휙
  • 같은 타깃 이틀 연속 = 무조건 금지(채근 0) → 반복 노출 불가
  • 부모 입장: 매일 다른 숙제 → 서사·안정감 없음, 뭘 꾸준히 해야 할지 모호
  • 재노출 과학(주 2~3회 같은 음식)과 정면 충돌

제안 — 주간 닻 + 일간 실행

  • 이번 주 미션 = "콩류와 친해지기" 하나. 월·수·금에 콩류, 다른 날은 변주
  • 같은 타깃이 여러 날 = 설계된 반복 노출(휙휙 아님)
  • 부모 입장: "이번 주는 이것" — 한 주 내내 같은 방향, 매일 표현만 다름
  • 주 1회 '좀 더 밀기'(채근)를 허용 = 이사님 정책

핵심 통찰: '매일 분석'은 반응성은 좋지만 안정성이 없고, '주간 고정'은 안정적이지만 그날 긴급 신호(전환·적신호)를 놓친다. 둘을 층으로 분리한다 — 주간이 방향을 잡고, 일간이 그 안에서 반응한다.

2. 2층 케이던스 아키텍처

크론은 지금처럼 하나(/api/cron/coach, 0 17 * * * = KST 02시). Vercel Hobby 2-크론 한도를 지키기 위해 새 크론을 추가하지 않고 라우트 안에서 KST 요일로 분기한다.

일요일(dow=0)주간 종합 + 회고
① 자녀 루프 진입 전 runWeeklyPlanning() — LLM 1콜로 지난 28일 분석 → weekly_plans upsert(닻 = 타깃·타깃 풀·보조 축·예산·아크 주차). 요일별 슬롯은 안 짠다. ② 일요일 편지 = '지난 주 회고 + 다음 주 미션' 프레임(weekly-review 시나리오, target/move 없음·PRAISE_ONLY 계열). ③ 월 첫 일요일은 여기에 포트폴리오 감사(trailing-84) 한 분기 더 — 월간 점검(§11).
월~토(dow=1~6)닻 안에서 일간 phase 결정
weekly_plans에서 닻(타깃·예산)을 읽고, 이번 주 누적(week-to-date) 데이터로 오늘 phase(노출/공고화/축하/휴식/안전)·무브·예산 소비를 매일 결정. 타깃은 주간 고정, HOW·WHEN만 일간이 그날 데이터로(§4).
# /api/cron/coach — 요일 분기 (단일 크론, 새 크론 0) const dow = kstDow(today) # 0=일 … 6=토 if (dow === 0) { # 일요일 = 주간 종합 for child of active: runWeeklyPlanning(child) → weekly_plans upsert # LLM 1콜/자녀 편지 = composeLetter({ scenario:'weekly-review', recap, mission }) } else { # 월~토 = 닻 안에서 일간 결정 for child of active: wp = loadWeeklyPlan(child, weekKey) # 없으면 planFor 폴백 wtd = weekToDate(child, weekStart) # 이번 주 월~어제 (토→월~금 5일) hist = trailing28(child) # 다주 노출 누적·distinct dish phase = decidePhase(wtd, hist, wp.budget, wp.ledger, dow) # 노출/공고화/축하/휴식/안전 plan = planFor({ weekly: wp, phase, signals }) # 타깃 고정·무브만 회전·예산 적기 소비 편지 = composeLetter({ ...plan }) # 작문·안전·유사도 가드 그대로 }

호환성: composeLetter·안전가드·유사도가드는 손대지 않는다. precomputed(plan)만 '매일 타깃 재선택'→'주간 고정 타깃 + 일간 phase'로 바뀐다. planFor는 폐기하지 않고 닻 부재 시 폴백으로 남긴다(점진 배포·무회귀).

미션 개수 · 수명 · '완료'의 의미 설계 결정

핵심 전제 — 단위는 '주당 음식 수'가 아니라 '음식당 주 수'다. 새 음식 1개 수용엔 8~10회 노출(까다로운 아이 15~20회)이 격일~주 2~3회 간격으로 필요 → 음식당 보통 3~5주. 한 주엔 2~3회만 적립하므로 한 주로는 한 음식의 노출 예산을 절대 못 채운다.

질문결정
미션 몇 개?부모 체감 1개 = 엔진 내부 1 + 0.5. 주력 식품 1개(주 2~3회 노출) + 보조 1개 — 보조는 식품이 아니라 다른 축(환경·자율성). 식품 타깃 2개 동시 = 금지("한 번에 하나" 위반·반복 노출 흩어짐 → 옛 '휙휙' 회귀). 상한 1.5 · 하한 1.
미션 수명미션 ≠ 한 주에 끝내는 과제. "이번 주에 이 음식에 2~3회 적립" → 같은 미션을 2~3주+ 지속(다주 아크). '주'는 미션 수명이 아니라 재진단 주기.
'완료' 정의이벤트가 아니라 누적 상태. 1~2회 먹음 ≠ 완료(변동성 범위 안·내리면 수용 붕괴). 판정 = distinct dish 2+(마스터리 배지 기준) 또는 연속 2~3회 거부 없는 수용. 판정은 일요일 회고 권한, 평일은 미션 못 끔.
주중 조기 수용미션 종료 ❌ → 유지(공고화) 모드로 격하: 그날 축하(행동 0) → 남은 날 push 끄고(채근 예산 선소진) 변주·칭찬으로 노출만 유지. 새 하드 미션 끼워넣기 = 안티패턴(축하 직후 새 숙제는 강화 파괴). 부모에겐 "정복/끝" 금지 → "친해지기 시작".

채근(주 1회) ≠ 노출(주 2~3회). 두 축을 혼동해 '주 1회만 노출'로 줄이면 노출 예산 부족으로 오히려 너무 느려진다. 노출(예산 배정)은 2~3회, 적극적 들이밀기(push 무브 mix/beside)만 1회.

4. 주간 ↔ 일간 역할 분담 — 그날 데이터 반응 + '토요일 문제'

핵심 경계선: 주간은 무엇을(WHAT) 잠그고, 일간은 어떻게·언제(HOW·WHEN)를 매일 그날 데이터로 결정한다. 요일별 슬롯(고정 대본)은 만들지 않는다 — 그게 일간의 반응성을 죽이기 때문(이 보고서 초안의 결함이었다).

초점 타깃 1개 · 타깃 풀 · 보조 축🔒 잠금—
예산(노출 2~3 · 채근 1 · 케이던스 간격) · 졸업 판정권 · 다주 메모리🔒 발급·독점—

"그럼 주간이 무용해지는 것 아닌가?" — 아니다(단 조건부). 주간이 'target 한 줄 + 카운터'만 주면 진짜로 '타깃 힌트 붙은 일간 엔진'으로 붕괴한다. 주간이 살아있는 건 일간이 구조적으로 못 보는 것을 쥐기 때문 — ① 다주 메모리(일간은 이번 주만 봄 → '콩류 3주째 7/10회 노출'은 주간 닻에만 존재) ② 케이던스 잠금(격일~주 2~3회는 주 단위 분배 → 일간이 매일 자유 판단하면 연타·권태) ③ 졸업 판정 독점(1~2회 먹음은 변동성 → 평일이 끄면 수용 붕괴). 이 셋이 빠지면 붕괴 맞다. 그래서 주간 = 일간의 행동 경계를 정의하는 헌법.

'토요일 문제' 해소 — 이중 데이터 창

현행 결함: 데이터 창이 trailing-7(어제까지 7일)이라 일요일 종합이 사실상 '지난주'만 본다 → 토요일에 월~금 5일이 0% 반영. (슬롯 미리 박기도 같은 병.) 해결 = 창을 둘로 분리.

week-to-date (주 시작 월 ~ 어제, 가변 길이) # 이번 주 진행 전담 → 오늘 phase · 예산 소비 · 무브 결정. 토요일엔 월~금 5일이 전부 들어온다. trailing-28 (어제 기준 28일 고정) # 다주 누적 전담 → 누적 노출 횟수 · distinct dish · 재노출 적기. 노출 이력 절대 보존.

토요일의 '오늘 뭐 할지'는 일요일이 박은 게 아니라 토요일 새벽에 이번 주 누적을 직접 보고 내린다. 슬롯이 없으니 미리 박힌 결정이 없다.

그래도 휙휙이 재발 안 하는 이유: 매일 재계산하는 건 HOW(phase·move·timing)뿐 — WHAT(타깃)은 주간 고정이라 일간 입력이 아니다. 게다가 phase는 누적의 단조(monotone) 함수(노출 횟수는 줄지 않음 → 노출일→공고화일→축하일로 전진만, 진동 없음).

5. ⭐ 채근 정책 변경 — 주 0회 → 주 1회

이사님 지시: "채근도 이제는 아예 안 하기보다 주당 1회 정도는 하는 걸로 하자." 이 한 줄이 옛 plan 엔진의 핵심 가드를 뒤집는다.

옛 규칙 (plan 엔진)

  • planFor: 어제와 같은 타깃이면 무조건 다른 각도(정체기)로 전환
  • = 같은 걸 이틀 연속 미는 채근 0회
  • 부작용: 반복 노출이 구조적으로 불가능(같은 음식 주 2~3회를 못 함)

새 규칙 (주간 닻)

  • 예산으로 노출 2~3회 배정(요일은 일간이 데이터로) = 반복 노출 = 정상
  • '직접 밀기(push)'는 ledger.pushUsed로 정확히 주 1회
  • 나머지 날은 같은 타깃이라도 부드러운 변주(조리법·궁합·환경·칭찬)

'채근(push)'의 정의 — 코드 판정

채근 = 같은 타깃을 적극적으로 들이미는 행동. 구체적으로 moveKey ∈ {mix, beside}('좋아하는 음식에 섞기'·'옆에 곁들이기')로 같은 타깃을 한 번 더 제안하는 것. 나머지 무브(recook·pair·cook-together·model·sensory)와 환경/자율성/칭찬은 변주이지 채근이 아니다.

① 요약은 raw를 대체하지 않는다병원 차트 원칙
meal_logs = 원본 랩(영구 단일 진실). 요약 = 읽기 캐시일 뿐, 항상 드릴다운 포인터(exposeCount·acceptCount·원천 log_date)를 동봉. 우리 실패 모드는 데이터 부족이 아니라 '약한 신호를 강한 판단으로 뭉갬'(카레유령=거부메모를 가짜 전환으로 날조·'1회 수용≠완료') — 요약이 "콩류 수용 시작"으로 뭉개면 월간이 그 아래 '사실 한 번뿐'을 못 본다. raw 동봉이 거짓 확신을 막는다.
② 상위 = 하위의 순수 함수(fold), 독립 재판단 금지계층 일관성
주간은 일간 기록들을 fold해 파생될 뿐 raw를 독립 재판단하지 않는다 → 층끼리 어긋날 입구가 없다. (지금은 일간 7일창·주간 28일창·차트 365일창이 각자 raw를 따로 집계해 창-경계 불일치가 구조적으로 가능.) transition/exposure 정의는 lib 한 곳에서 import.

과설계 판정 + MVP

지금(MVP) — 거의 공짜·신규 테이블 0

  • (A) 이미 매일 써지는 period_summaries.week 행에 엔진 판단 3필드(targetsTried[]·exposeCount{음식:n}·verdict)를 daily plan들을 fold해 추가(~30줄)
  • (B) 편지 엔진이 그 week 행을 닻으로 read — 'write-only dead-end'를 깨는 핵심 한 줄. → 28일 밖 누적 노출 아크(8~10회, 까다로우면 15~20)를 plan이 참조
  • raw 동봉으로 lossy 방어. 진짜 실익 = 장기 노출 아크 하나(비용 절감 논거는 4명에선 약함)

나중(스케일 후) — 보류

  • 전향 weekly_plans(mission_target·budget·졸업판정)
  • 월간 포트폴리오 전환·증분 롤업·dirty 캐시 무효화
  • L2가 raw 대신 weekly_summary 읽기로 전환(비용 ROI는 자녀 수백·노출 누적 시점)
  • 트리거 = 활성 수십~수백 명 + 365일 raw 재집계가 느려질 때

왜 지금 풀 3층 롤업은 과설계: '정확히 판단'의 진짜 병목은 메모리 구조가 아니라 데이터 부족(활성 ~4명)이다. 이 규모에선 윈도우 raw 재집계가 더 단순하고 똑같이 정확하다(재집계 비용 무시 가능). 방향은 옳다 — 후향 요약(경과기록)부터 적립하고, 데이터가 쌓이면 전향 처방(작전)을 그 위에 얹는다.

13. 행동 지연 · 일요일 주경계 · 미션 비노출 이사님 결정

부모 대면 동작과 엔진 신호 해석을 바꾸는 세 결정. 공통 축 = "부모가 보는 건 매일 코치 편지 하나뿐" + "조언과 실행 사이엔 시차가 있다".

A. 미션 비노출 — 코치 편지로만 움직인다 확정

미션('이번 주 콩류')은 엔진 내부 나침반일 뿐, 부모에겐 절대 노출 안 한다. '이번 주 미션' 카드·진척 게이지·숙제 목록·N/10회 노출 같은 카운트 전부 0. 부모가 보는 건 매일 따뜻한 편지 하나 + 일요일 회고.

누수 경로차단
편지가 '이번 주 미션/목표/과제/숙제'라고 받아씀letterDeterministicBad에 정규식 추가(미션·목표·과제·숙제·'N회 노출'·'사세요/구매/장 보세요' 차단) + promptHint 금지어
RLS 누수: 부모 클라가 weekly_plans에서 mission_target 직접 fetchparent-read 정책에서 mission_target·target_pool·ledger 컬럼 select 제외(안전 뷰). 부모는 편지 텍스트만.
도감 '추천' 배지로 이번 주 타깃 역추론도감 추천은 빈도/영양 기준 독립 — 코칭 타깃과 분리(현행 유지)

B. 행동 지연(procurement lag) — 시차를 엔진이 인지 신규 설계

편지를 오늘 봐도 식재료 사서 차리기까지 며칠. 이 시차를 무시하면 우리 실패 모드(약신호를 강판단으로 뭉갬·카레유령)가 재발한다 — "타깃 안 나타남"을 거부로 오판하거나, 살 시간도 없는데 "왜 안 했어요" 채근. 결정론 규칙 6개(신규 테이블 0·ledger 키 3개 추가):

#규칙
R1노출 카운트 = 실제 차림만. exposeCount는 meal_logs.ingredients에 타깃이 실제 등장한 끼니만 +1. 조언·편지 발행은 노출 0(조언≠노출). 8~10회 아크가 '엔진이 N번 권함'이 아니라 '식탁에 N번 차려짐'으로 정직하게 전진.
R2firstServe 게이트. 이번 주 타깃이 한 번도 안 차림(firstServeDow=null)이면 push·예산 페이싱·채근 멘트 전부 잠금. 못 산 걸 들이밀 대상 자체가 없다 → push 0회.
R3absence ≠ refusal. '안 나타남'=noExposure(중립), 거부는 refused/ate_well:false만. grace는 '거부 판정 유예'가 아니라 push 보류 게이트 — 차렸는데 거부면 grace 무관 즉시 인정(진짜 거부 안 놓침).
R4grace = 3~4일(주말 장 + 평일 장 기회). adviceGivenAt(타깃 첫 권유일)부터 카운트. firstServe 발생 시 조기 종료(달력은 안전 상한일 뿐) → 일찍 산 부모는 일찍 신호, 늦게 산 부모는 채근 안 당함.
R5아크는 progressWeek(노출 진척주) 기준, 달력주 아님. arc_week++는 노출이 실제 누적된 주만. 시차로 달력 6~10주여도 진척주 3~5 유지 → 월간 stuck/abandon이 '미조달 ≠ stuck'을 구분(거짓 abandon 방지).
R0즉시 실행 우선(시차 0). 집에 이미 있는 것(favoriteFoods·최근 등장·박스 구성품) 먼저 소비. 새로 살 타깃은 일요일 리드타임으로 안내하고, 그 사이 평일은 보유 식재료로 아크를 굴림.

C. 일요일 주경계 + 리드타임 Q1 확정

이사님: "월요일 오전은 바빠서 못 본다. 엄마가 일요일에 우리 편지 보고 다음 주 계획 세워야지." → 주 경계를 일요일로.

일요일 회고(지난 주) + 다음 주 안내(장보기 힌트) # 닻 write: week_key = isoWeekKey(today+1) ▼ 주말 장보기 = 시차 1차 방어선(월요일부터 집에 식재료) 월~토 닻 실행(자기 날짜 키로 read) # grace = 시차 2차 방어선

"닻 쓰기(일요일)만 +1, 읽기(월~토)는 자기 날짜." isoWeekKey는 월요일 시작이라 일요일을 그냥 키하면 지난주가 나옴 → isoWeekKey(today+1)로 다가올 주 키 저장(off-by-one 해결). 일요일 편지 보관 3→7일(월요일 늦게 봐도 안내 살아있게).

D. 편지 UX — 권유이지 과제가 아니다 확정

일요일 편지

  • 회고 win 1개(새로 친해진 음식·distinct dish 늘어난 것 — 진척 숫자 칭찬 금지)
  • 다음 주 권유 1줄 — 가정형: "주말에 두부 한 모 있으면, 이번 주에 ○○이 좋아하는 볶음밥에 같이 넣어볼 수 있어요"
  • 주어 = "코치가 ~해볼게요"(엄마 과제 아님). 명령형 '사세요' 금지

평일 편지

  • 이미 집에 있는 것 우선(오늘 당장 할 수 있는 것)
  • 새로 살 것은 '다음 장 볼 때' 미리·부드럽게
  • grace 안엔 '왜 안 했어요' 채근 금지 — 보유 식재료 변주·환경 축으로 환기
  • '이번 주 ~하기' 같은 주간 과제어 금지. 그날 한 걸음만

죄책감 0 경계: 권유(있으면·천천히·여유될 때)=OK / 과제(이번 주 안에·꼭·N회)=금지. 못 샀을 때 다음 편지가 추궁 안 함(R2가 firstServe=null이면 채근 멘트 데이터로 차단). 장보기 힌트는 ~주 1회 로테이션(매일 반복 X).

과설계 경계: 신규 테이블·크론·부모 UI 0. ledger jsonb 키 3개(adviceGivenAt·firstServeDow·progressWeek) + decidePhase firstServe 분기 ~10줄 + letterDeterministicBad 정규식 3종. 활성 ~4명에선 '관점만 심고' 복잡한 조달 예측은 보류.

14. 주간 종합의 깊이 — 비싼 모델·종합 신호·35이론·의사 소견 이사님 결정

이사님: "주간 계획은 비싼 모델로 종합적으로 봐서 가이드를 잡으면 좋겠다. 식단 말고도 35개 국제 편식 이론 기반으로 검토하나? 차트 남길 때 LLM으로 의사처럼 종합 소견을 남기나?" → 네 보강은 하나의 원칙으로 묶인다: 주간 = 비싼 모델이 모든 신호를 35이론 렌즈로 읽어 의사식 소견 + 방향을 남기는 회진(回診). 일간은 그 소견을 들고 그날 한 걸음을 싸게 실행. 현재는 편지가 Haiku 단일·주간 종합 미구현이라 전부 미구현 설계.

① 모델 티어 — 비싼 판단은 드물게, 싼 실행은 자주

층모델빈도역할
일간결정론 + Haiku(작문만)매일닻 안에서 그날 한 걸음·톤
주간 종합Opus급(비싼)자녀당 주 1회종합 회진·방향·소견
월간 감사Opus급 또는 결정론월 1회포트폴리오·전략(§11)

자녀당 주 1회뿐이라 비싼 모델 비용 OK(활성 ~4명이면 Opus 주 4콜=무해). 가장 레버리지 큰 결정(이번 주 방향)에 가장 똑똑한 모델을 쓴다. 이게 "LLM 종합 판단은 주간에 적소, 일간은 결정론" 원칙의 모델 티어 구현.

② 종합 신호 — "어떤 음식 먹었냐"는 입력 하나일 뿐

주간 종합이 보는 것: 식단(무엇/거부/전환) + 구조화 입력(식감·자율성·식사환경·식사시간) + 만성질환 경과(변비·철결핍 등) + 성장(키·몸무게/BMI 추세) + ICFQ 위험·재노출 적기 + 35이론 렌즈(아래). 식단만 보던 게 아니라 아이 상태 전체를 읽는다.

③ 35이론 렌즈 — 방법론 진단이 주간의 핵심 산출

35이론(정본: 시나리오 엔진에 13시나리오로 구현)은 지금 일간 시나리오 선택에만 쓰인다. 보강: 주간 종합이 "이 아이의 지금 패턴에 어느 이론/방법론이 맞나"를 비싼 모델로 진단해 이번 주 미션·접근을 잡는다. 예 — 신공포 단계면 SOS 감각 사다리, 자율성 다툼이면 Satter 식사 분담, 정체면 푸드체이닝. 식단 숫자가 아니라 방법론 진단이 주간의 산출.

④ 의사식 종합 소견 — 차트에 narrative impression

병원 차트가 랩 수치 + 의사 impression을 둘 다 남기듯(§12 철칙), 주간 종합이 LLM으로 '경과 소견'을 차트에 적는다 — 숫자가 아니라 판단.

예시 소견(weekly_summary.impression)
"지난주 콩류 노출 3→5회로 늘었으나 형태는 두부 한 가지에 머묾(distinct=1). 화·목 자율성 다툼 메모 2건 — 신공포보다 통제 욕구 신호. 성장 곡선 정상 추세. 이번 주는 식감보다 식사 분위기·선택권 우선(Satter), 콩류는 형태만 바꿔 가볍게 유지."

이 소견을 일간 Haiku(편지 톤·방향 일관)와 다음 주 종합(longitudinal·전주 소견 대비)이 읽는다. 숫자=raw 드릴다운, 소견=판단 — 둘 다 보존(§12). 미션 비노출(§13): 소견은 엔진 내부용, 부모에겐 편지로만.

구현 위치: 주간 종합층(미구현)을 지을 때 함께. runWeeklyPlanning이 Opus로 종합 신호+35이론을 읽어 weekly_plans.impression(소견)·mission·budget을 산출. 일간/월간은 read. 신규 비용 = 자녀당 주 1 Opus 콜.
