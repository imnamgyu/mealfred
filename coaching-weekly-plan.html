코칭 주간 계획 엔진 — 2층 케이던스 설계

설계 문서 · 2026-06-09 · 주간 닻(일요일 LLM 종합) + 일간 결정론 실행. 후속 설계: plan 엔진(배포)·시나리오 엔진의 위에 '한 주'라는 시간축을 얹는다. 이 문서는 설계만 — 코드 변경 없음.

🎯 한눈에

  • 문제: 코칭 크론이 매일 새벽 최근 7일을 새로 보고 시나리오·타깃을 즉석 산출 → 며칠 만에 콩류→생선→과일로 '휙휙' 바뀐다. 편식 개선은 같은 음식 주 2~3회 반복 노출이 핵심인데, 매일 다른 숙제는 그 과학과 어긋난다. 주간 닻이 없다.
  • 해법(역할 분담): 일요일 새벽 2시 = LLM이 지난 28일을 종합해 무엇을(WHAT) — 초점 타깃·타깃 풀·예산(노출 2~3·채근 1·간격)만 잠근다(weekly_plans 1행, 요일 슬롯 아님). 월~토 새벽 2시 = 코드가 닻 안에서 어떻게·언제(HOW·WHEN)를 그날 데이터로 매일 결정(오늘 phase·무브·예산 소비). → 안정은 주간(타깃 고정), 반응은 일간(매일 phase). 비용 1/7.
  • ⭐ 채근 정책 변경(이사님): 옛 '같은 타깃 이틀 연속 금지'(=채근 0회) → 주 1회는 같은 걸 좀 더 민다. 예산으로 노출 2~3회(반복 노출) + '직접 밀기(push)'는 ledger.pushUsed로 정확히 주 1회 하드 캡. ARFID 의심·휴면 복귀 주는 push 선잠금(0회).
  • 토요일 문제 해소(§4): 데이터 창을 이중으로 — week-to-date(이번 주 월~어제)로 오늘을 정하고 trailing-28로 다주 노출을 보존. 토요일엔 월~금 5일이 그대로 반영된다. 타깃은 주간 고정이라 매일 재계산해도 '휙휙' 재발 안 함(HOW만 데이터 반응).
  • 안전 제1원칙: 일간 결정론 경로는 주간 닻 유무와 무관하게 항상 단독으로 편지를 낼 수 있어야 한다. 닻 없음/LLM 실패 → 현행 planFor 폴백(무편지 0). 닻은 '있으면 좋은 안정 장치'.
  • 설계 범위: 4개 에이전트 병렬 설계 + 적대 엣지케이스 10종 검증. 결정 필요 5건(§9) — 주 경계·일요일 부하·미션 부모 노출·채근 톤 적대검수·구조화 입력 신뢰도.

1. 왜 주간 계획인가 — '매일 재계산'의 한계

현재 cron/coach는 매일 새벽 2시(KST) planFor()가 최근 7일 윈도우를 처음부터 새로 보고 그날의 시나리오·타깃·무브를 즉석에서 뽑는다. plan 엔진이 '하루 안에서의 중복'은 막았지만(시그니처 dedup·채근 방지), '한 주에 걸친 일관성'을 잡는 시간축이 없다.

현재 — 매일 독립 재계산

  • 월: 콩류 · 화: 생선 · 수: 과일 · 목: 콩류… 타깃이 휙휙
  • 같은 타깃 이틀 연속 = 무조건 금지(채근 0) → 반복 노출 불가
  • 부모 입장: 매일 다른 숙제 → 서사·안정감 없음, 뭘 꾸준히 해야 할지 모호
  • 재노출 과학(주 2~3회 같은 음식)과 정면 충돌

제안 — 주간 닻 + 일간 실행

  • 이번 주 미션 = "콩류와 친해지기" 하나. 월·수·금에 콩류, 다른 날은 변주
  • 같은 타깃이 여러 날 = 설계된 반복 노출(휙휙 아님)
  • 부모 입장: "이번 주는 이것" — 한 주 내내 같은 방향, 매일 표현만 다름
  • 주 1회 '좀 더 밀기'(채근)를 허용 = 이사님 정책

핵심 통찰: '매일 분석'은 반응성은 좋지만 안정성이 없고, '주간 고정'은 안정적이지만 그날 긴급 신호(전환·적신호)를 놓친다. 둘을 층으로 분리한다 — 주간이 방향을 잡고, 일간이 그 안에서 반응한다.

2. 2층 케이던스 아키텍처

크론은 지금처럼 하나(/api/cron/coach, 0 17 * * * = KST 02시). Vercel Hobby 2-크론 한도를 지키기 위해 새 크론을 추가하지 않고 라우트 안에서 KST 요일로 분기한다.

일요일(dow=0)주간 종합 + 회고
① 자녀 루프 진입 전 runWeeklyPlanning() — LLM 1콜로 지난 28일 분석 → weekly_plans upsert(닻 = 타깃·타깃 풀·보조 축·예산·아크 주차). 요일별 슬롯은 안 짠다. ② 일요일 편지 = '지난 주 회고 + 다음 주 미션' 프레임(weekly-review 시나리오, target/move 없음·PRAISE_ONLY 계열).
월~토(dow=1~6)닻 안에서 일간 phase 결정
weekly_plans에서 닻(타깃·예산)을 읽고, 이번 주 누적(week-to-date) 데이터로 오늘 phase(노출/공고화/축하/휴식/안전)·무브·예산 소비를 매일 결정. 타깃은 주간 고정, HOW·WHEN만 일간이 그날 데이터로(§4).
# /api/cron/coach — 요일 분기 (단일 크론, 새 크론 0) const dow = kstDow(today) # 0=일 … 6=토 if (dow === 0) { # 일요일 = 주간 종합 for child of active: runWeeklyPlanning(child) → weekly_plans upsert # LLM 1콜/자녀 편지 = composeLetter({ scenario:'weekly-review', recap, mission }) } else { # 월~토 = 닻 안에서 일간 결정 for child of active: wp = loadWeeklyPlan(child, weekKey) # 없으면 planFor 폴백 wtd = weekToDate(child, weekStart) # 이번 주 월~어제 (토→월~금 5일) hist = trailing28(child) # 다주 노출 누적·distinct dish phase = decidePhase(wtd, hist, wp.budget, wp.ledger, dow) # 노출/공고화/축하/휴식/안전 plan = planFor({ weekly: wp, phase, signals }) # 타깃 고정·무브만 회전·예산 적기 소비 편지 = composeLetter({ ...plan }) # 작문·안전·유사도 가드 그대로 }

호환성: composeLetter·안전가드·유사도가드는 손대지 않는다. precomputed(plan)만 '매일 타깃 재선택'→'주간 고정 타깃 + 일간 phase'로 바뀐다. planFor는 폐기하지 않고 닻 부재 시 폴백으로 남긴다(점진 배포·무회귀).

3. weekly_plans — 주간 닻 (데이터모델)

자녀 × ISO주 1행. 일요일 종합이 쓰고(service_role) 월~토가 읽는다. plan 엔진의 context.plan 원장을 '하루'에서 '한 주'로 끌어올린 것.

미션 개수 · 수명 · '완료'의 의미 설계 결정

핵심 전제 — 단위는 '주당 음식 수'가 아니라 '음식당 주 수'다. 새 음식 1개 수용엔 8~10회 노출(까다로운 아이 15~20회)이 격일~주 2~3회 간격으로 필요 → 음식당 보통 3~5주. 한 주엔 2~3회만 적립하므로 한 주로는 한 음식의 노출 예산을 절대 못 채운다.

질문결정
미션 몇 개?부모 체감 1개 = 엔진 내부 1 + 0.5. 주력 식품 1개(주 2~3회 노출) + 보조 1개 — 보조는 식품이 아니라 다른 축(환경·자율성). 식품 타깃 2개 동시 = 금지("한 번에 하나" 위반·반복 노출 흩어짐 → 옛 '휙휙' 회귀). 상한 1.5 · 하한 1.
미션 수명미션 ≠ 한 주에 끝내는 과제. "이번 주에 이 음식에 2~3회 적립" → 같은 미션을 2~3주+ 지속(다주 아크). '주'는 미션 수명이 아니라 재진단 주기.
'완료' 정의이벤트가 아니라 누적 상태. 1~2회 먹음 ≠ 완료(변동성 범위 안·내리면 수용 붕괴). 판정 = distinct dish 2+(마스터리 배지 기준) 또는 연속 2~3회 거부 없는 수용. 판정은 일요일 회고 권한, 평일은 미션 못 끔.
주중 조기 수용미션 종료 ❌ → 유지(공고화) 모드로 격하: 그날 축하(행동 0) → 남은 날 push 끄고(채근 예산 선소진) 변주·칭찬으로 노출만 유지. 새 하드 미션 끼워넣기 = 안티패턴(축하 직후 새 숙제는 강화 파괴). 부모에겐 "정복/끝" 금지 → "친해지기 시작".

채근(주 1회) ≠ 노출(주 2~3회). 두 축을 혼동해 '주 1회만 노출'로 줄이면 노출 예산 부족으로 오히려 너무 느려진다. 노출(예산 배정)은 2~3회, 적극적 들이밀기(push 무브 mix/beside)만 1회.

4. 주간 ↔ 일간 역할 분담 — 그날 데이터 반응 + '토요일 문제'

핵심 경계선: 주간은 무엇을(WHAT) 잠그고, 일간은 어떻게·언제(HOW·WHEN)를 매일 그날 데이터로 결정한다. 요일별 슬롯(고정 대본)은 만들지 않는다 — 그게 일간의 반응성을 죽이기 때문(이 보고서 초안의 결함이었다).

결정 대상주간(느린 축·일 1회)일간(빠른 축·매일)
초점 타깃 1개 · 타깃 풀 · 보조 축🔒 잠금—
예산(노출 2~3 · 채근 1 · 케이던스 간격) · 졸업 판정권 · 다주 메모리🔒 발급·독점—
오늘 phase(노출/공고화/축하/휴식/안전)—✅ 매일
무브 · 각도 · 음식 · 문장 · 예산 적기 소비—✅ 매일

"그럼 주간이 무용해지는 것 아닌가?" — 아니다(단 조건부). 주간이 'target 한 줄 + 카운터'만 주면 진짜로 '타깃 힌트 붙은 일간 엔진'으로 붕괴한다. 주간이 살아있는 건 일간이 구조적으로 못 보는 것을 쥐기 때문 — ① 다주 메모리(일간은 이번 주만 봄 → '콩류 3주째 7/10회 노출'은 주간 닻에만 존재) ② 케이던스 잠금(격일~주 2~3회는 주 단위 분배 → 일간이 매일 자유 판단하면 연타·권태) ③ 졸업 판정 독점(1~2회 먹음은 변동성 → 평일이 끄면 수용 붕괴). 이 셋이 빠지면 붕괴 맞다. 그래서 주간 = 일간의 행동 경계를 정의하는 헌법.

'토요일 문제' 해소 — 이중 데이터 창

현행 결함: 데이터 창이 trailing-7(어제까지 7일)이라 일요일 종합이 사실상 '지난주'만 본다 → 토요일에 월~금 5일이 0% 반영. (슬롯 미리 박기도 같은 병.) 해결 = 창을 둘로 분리.

week-to-date (주 시작 월 ~ 어제, 가변 길이) # 이번 주 진행 전담 → 오늘 phase · 예산 소비 · 무브 결정. 토요일엔 월~금 5일이 전부 들어온다. trailing-28 (어제 기준 28일 고정) # 다주 누적 전담 → 누적 노출 횟수 · distinct dish · 재노출 적기. 노출 이력 절대 보존.

토요일의 '오늘 뭐 할지'는 일요일이 박은 게 아니라 토요일 새벽에 이번 주 누적을 직접 보고 내린다. 슬롯이 없으니 미리 박힌 결정이 없다.

그래도 휙휙이 재발 안 하는 이유: 매일 재계산하는 건 HOW(phase·move·timing)뿐 — WHAT(타깃)은 주간 고정이라 일간 입력이 아니다. 게다가 phase는 누적의 단조(monotone) 함수(노출 횟수는 줄지 않음 → 노출일→공고화일→축하일로 전진만, 진동 없음).

토휴식/유지WTD=월~금 5일 전부 봄(노출2·수용1·push 소진·가지거부1) → 변주·칭찬

현행이면 토요일도 trailing-7로 콩을 또 밀거나(휙휙), 슬롯 모델이면 일요일에 박은 '환경' 슬롯을 데이터 무시하고 실행. 예산 모델은 화/축하·목/새거부·금/예산소비가 전부 그날 데이터로 갈린다.

일관성 · 충돌 우선순위

dormancy(휴면 복귀) > shock(데이터 쇼크) > ARFID 게이트 > 닻 타깃 준수 > planFor 자율(폴백)

닻(타깃·예산) 전체 재작성은 오직 다음 일요일. 월~토는 닻 안에서 phase만 고르고 타깃을 못 바꾼다. 예외 — 주중 결핍 구조가 근본 변동(red 2개+ 신규 && 타깃 풀이 하나도 안 다룸 && 이번 주 첫 발생)이면 남은 주의 풀만 갱신(과거 노출 카운트·pushUsed 보존). ledger.reanchorUsed로 주 1회 하드캡 → 재닻 남발=옛 휙휙 차단. 단순 새거부 1건은 재닻 아님(편지 하단 한 줄).

5. ⭐ 채근 정책 변경 — 주 0회 → 주 1회

이사님 지시: "채근도 이제는 아예 안 하기보다 주당 1회 정도는 하는 걸로 하자." 이 한 줄이 옛 plan 엔진의 핵심 가드를 뒤집는다.

옛 규칙 (plan 엔진)

  • planFor: 어제와 같은 타깃이면 무조건 다른 각도(정체기)로 전환
  • = 같은 걸 이틀 연속 미는 채근 0회
  • 부작용: 반복 노출이 구조적으로 불가능(같은 음식 주 2~3회를 못 함)

새 규칙 (주간 닻)

  • 예산으로 노출 2~3회 배정(요일은 일간이 데이터로) = 반복 노출 = 정상
  • '직접 밀기(push)'는 ledger.pushUsed로 정확히 주 1회
  • 나머지 날은 같은 타깃이라도 부드러운 변주(조리법·궁합·환경·칭찬)

'채근(push)'의 정의 — 코드 판정

채근 = 같은 타깃을 적극적으로 들이미는 행동. 구체적으로 moveKey ∈ {mix, beside}('좋아하는 음식에 섞기'·'옆에 곁들이기')로 같은 타깃을 한 번 더 제안하는 것. 나머지 무브(recook·pair·cook-together·model·sensory)와 환경/자율성/칭찬은 변주이지 채근이 아니다.

주 1회 강제 — 결정론 카운트

장치규칙
상한(주 1회)주간 예산에 push:1 + pushWindow(화~목) 발급. 일간이 적기(노출 2회+ 누적 && 수용 징후)에 1회 소비 — pushUsed=false면 적용 + pushUsed=true·pushDay=today 기록. 이미 true면 강등(moveKey를 push계열→비push계열로, 타깃은 유지=반복 노출 지속·들이밀기만 제거). 예산이 2여도 코드가 1회로 클램프.
하한(0 허용)그 주 적기를 못 만나거나(수용 징후 없음) 안전 인터럽트에 밀리면 그 주 0회 — '최대 1회'이지 '의무 1회'가 아니다. 금요일 고정이 아니라 적기에 떨어진다(§4 워크스루).
0회 강제잠금ARFID 의심(icfqRiskCount≥2·증상 메모·먹는 종류 감소) 또는 휴면 복귀 주는 종합 시점에 pushUsed=true 선잠금 → 그 주 채근 0회. 압박이 이탈·역효과인 아이를 보호.

왜 DB 컬럼 하나로 충분한가: 채근 카운트가 weekly_plans.ledger.pushUsed 한 곳에만 있어(단일 진실), 일일 재실행(force/QA)에도 멱등이고 별도 카운터가 필요 없다. '주 1회'가 데이터 구조에 박혀 자연히 강제된다.

시그니처 dedup은 유지. 같은 타깃이 여러 날 와도 (frame·move·도입)이 최근 3일과 겹치면 무브·도입을 회전 → 복붙 편지 방지. dedup의 차단 대상이 '의미(같은 처방)'에서 '표현(같은 문장틀)'으로 좁아진다(타깃 반복은 주간 미션이 책임지므로 허용).

8. 적대 엣지케이스 10종 + 결정론 안전망

2층을 깨려고 시도한 10개 케이스. 안전 제1원칙 = 일간 결정론은 닻 유무와 무관하게 항상 단독으로 편지를 낼 수 있어야 한다(닻은 '있으면 좋은' 안정 장치). 모든 완화책은 weekly_plans 1개 테이블 + planFor 폴백에 수렴한다.

9. 결정 필요 — 이사님 확인
