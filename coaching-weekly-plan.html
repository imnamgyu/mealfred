코칭 주간 계획 엔진 — 2층 케이던스 설계

설계 문서 · 2026-06-09 · 주간 닻(일요일 LLM 종합) + 일간 결정론 실행. 후속 설계: plan 엔진(배포)·시나리오 엔진의 위에 '한 주'라는 시간축을 얹는다. 이 문서는 설계만 — 코드 변경 없음.

🎯 한눈에

  • 문제: 코칭 크론이 매일 새벽 최근 7일을 새로 보고 시나리오·타깃을 즉석 산출 → 며칠 만에 콩류→생선→과일로 '휙휙' 바뀐다. 편식 개선은 같은 음식 주 2~3회 반복 노출이 핵심인데, 매일 다른 숙제는 그 과학과 어긋난다. 주간 닻이 없다.
  • 해법(2층 케이던스): 일요일 새벽 2시 = LLM이 지난 한 주를 종합해 다음 주 미션 1개 + 요일별 슬롯을 짠다(weekly_plans 1행). 월~토 새벽 2시 = 코드가 그날 슬롯을 결정론 실행하고 그날 데이터로 작은 교정만. → LLM 종합 판단은 '주간'에(적소), 일일은 결정론. 비용 1/7.
  • ⭐ 채근 정책 변경(이사님): 옛 '같은 타깃 이틀 연속 금지'(=채근 0회) → 주 1회는 같은 걸 좀 더 민다. 주간 미션 타깃을 슬롯에 2~3회 의도 배치(반복 노출) + '직접 밀기(push)'는 ledger.pushUsed로 정확히 주 1회 하드 캡. ARFID 의심·휴면 복귀 주는 push 선잠금(0회).
  • 일관성: 월~토는 시나리오를 다시 안 고른다(슬롯이 닻). 교정은 화이트리스트 override만 — 안전 인터럽트(전환 축하·적신호)·타깃 소멸·무브 회전. 닻 전체 재작성은 오직 다음 일요일.
  • 안전 제1원칙: 일간 결정론 경로는 주간 닻 유무와 무관하게 항상 단독으로 편지를 낼 수 있어야 한다. 닻 없음/LLM 실패 → 현행 planFor 폴백(무편지 0). 닻은 '있으면 좋은 안정 장치'.
  • 설계 범위: 4개 에이전트 병렬 설계 + 적대 엣지케이스 10종 검증. 결정 필요 5건(§9) — 주 경계·일요일 부하·미션 부모 노출·채근 톤 적대검수·구조화 입력 신뢰도.

1. 왜 주간 계획인가 — '매일 재계산'의 한계

현재 cron/coach는 매일 새벽 2시(KST) planFor()가 최근 7일 윈도우를 처음부터 새로 보고 그날의 시나리오·타깃·무브를 즉석에서 뽑는다. plan 엔진이 '하루 안에서의 중복'은 막았지만(시그니처 dedup·채근 방지), '한 주에 걸친 일관성'을 잡는 시간축이 없다.

현재 — 매일 독립 재계산

  • 월: 콩류 · 화: 생선 · 수: 과일 · 목: 콩류… 타깃이 휙휙
  • 같은 타깃 이틀 연속 = 무조건 금지(채근 0) → 반복 노출 불가
  • 부모 입장: 매일 다른 숙제 → 서사·안정감 없음, 뭘 꾸준히 해야 할지 모호
  • 재노출 과학(주 2~3회 같은 음식)과 정면 충돌

제안 — 주간 닻 + 일간 실행

  • 이번 주 미션 = "콩류와 친해지기" 하나. 월·수·금에 콩류, 다른 날은 변주
  • 같은 타깃이 여러 날 = 설계된 반복 노출(휙휙 아님)
  • 부모 입장: "이번 주는 이것" — 한 주 내내 같은 방향, 매일 표현만 다름
  • 주 1회 '좀 더 밀기'(채근)를 허용 = 이사님 정책

핵심 통찰: '매일 분석'은 반응성은 좋지만 안정성이 없고, '주간 고정'은 안정적이지만 그날 긴급 신호(전환·적신호)를 놓친다. 둘을 층으로 분리한다 — 주간이 방향을 잡고, 일간이 그 안에서 반응한다.

2. 2층 케이던스 아키텍처

크론은 지금처럼 하나(/api/cron/coach, 0 17 * * * = KST 02시). Vercel Hobby 2-크론 한도를 지키기 위해 새 크론을 추가하지 않고 라우트 안에서 KST 요일로 분기한다.

일요일(dow=0)주간 종합 + 회고
① 자녀 루프 진입 전 runWeeklyPlanning() — LLM 1콜로 지난 7일 분석 → weekly_plans upsert(미션 + 월~토 슬롯). ② 일요일 편지 = '지난 주 회고 + 다음 주 미션' 프레임(weekly-review 시나리오, target/move 없음·PRAISE_ONLY 계열).
월~토(dow=1~6)슬롯 결정론 실행 + 작은 교정
weekly_plans에서 그날 슬롯을 읽어 composeLetter에 주입. 시나리오·타깃은 슬롯이 결정(selectScenario 호출 안 함). 그날 데이터로 화이트리스트 override만(§5·§6).
# /api/cron/coach — 요일 분기 (단일 크론, 새 크론 0) const dow = kstDow(today) # 0=일 … 6=토 if (dow === 0) { # 일요일 = 주간 종합 for child of active: runWeeklyPlanning(child) → weekly_plans upsert # LLM 1콜/자녀 편지 = composeLetter({ scenario:'weekly-review', recap, mission }) } else { # 월~토 = 슬롯 실행 for child of active: wp = loadWeeklyPlan(child, weekKey) # 없으면 lazy 폴백 slot = wp.slots[dow] # {scenarioId,target,moveKey,push} plan = applySlotWithCorrection(slot, todaySignals) # 작은 교정 편지 = composeLetter({ ...plan }) # 작문·안전·유사도 가드 그대로 }

호환성: composeLetter·안전가드·유사도가드는 손대지 않는다. precomputed(plan)만 '일일 즉석 산출'→'슬롯 기반'으로 바뀐다. planFor는 폐기하지 않고 닻 부재 시 폴백으로 남긴다(점진 배포·무회귀).

3. weekly_plans — 주간 닻 (데이터모델)

자녀 × ISO주 1행. 일요일 종합이 쓰고(service_role) 월~토가 읽는다. plan 엔진의 context.plan 원장을 '하루'에서 '한 주'로 끌어올린 것.

-- sql/2026-06-09_weekly_plans.sql create table public.weekly_plans ( child_id uuid not null references children(id) on delete cascade, week_key text not null, -- isoWeekKey(일요일) — lib/progress.ts 재사용 status text default 'active', -- active|degraded|stale_carried|cold_synth source text not null, -- weekly_llm|carried|cold_synth|none mission text, -- 이번 주 미션 1줄 (부모용·편지마다 일관 주입) mission_target text, -- 미션의 핵심 타깃 (슬롯에 2~3회 배치) slots jsonb not null, -- {"1":{scenarioId,target,moveKey,push}, … "6":{…}} ledger jsonb, -- {pushUsed,pushDay,shockResetAt,missionTouch[]} ⭐채근 카운트 basis jsonb, -- 종합 근거 스냅샷(지난주 metrics·reds·favoriteFoods) basis_hash text, -- 7일 입력 지문(동일하면 LLM 스킵·carry) basis_attends_daycare boolean, -- 등원 변경 감지(§8 ⑧) generated_at timestamptz default now(), primary key (child_id, week_key) ); alter table public.weekly_plans enable row level security; create policy weekly_plans_read on public.weekly_plans for select using (child_id in (select id from children where parent_id = auth.uid()));
저장물역할
mission / mission_target이번 주 단일 방향. 모든 평일 편지 하단에 일관 주입 → 서사 연속. 일요일 회고 편지에 재사용.
slots (월~토 6칸)요일별 {시나리오·타깃·무브·push} 확정. 이게 곧 일일 — 월~토는 슬롯만 읽는다.
ledger.pushUsed⭐ 채근 1회/주의 단일 진실. DB 컬럼 하나로 하드 캡(코드가 절대 2회 못 넘김).
basis_hash지난주 입력이 같으면 일요일 LLM 스킵·carry(비용 절감). plan 엔진의 srcHash 패턴 동일.

스키마 변경 최소: 신규 테이블 1개 + coach_letters.context(기존 jsonb)에 weekKey·slotDow·fromSlot·slotOverride·pushApplied 키 추가(DDL 불필요). 새 RPC 불필요 — 크론이 직접 upsert.

미션 개수 · 수명 · '완료'의 의미 설계 결정

핵심 전제 — 단위는 '주당 음식 수'가 아니라 '음식당 주 수'다. 새 음식 1개 수용엔 8~10회 노출(까다로운 아이 15~20회)이 격일~주 2~3회 간격으로 필요 → 음식당 보통 3~5주. 한 주 슬롯은 2~3회만 적립하므로 한 주로는 한 음식의 노출 예산을 절대 못 채운다.

질문결정
미션 몇 개?부모 체감 1개 = 엔진 내부 1 + 0.5. 주력 식품 1개(주 2~3회 노출) + 보조 1개 — 보조는 식품이 아니라 다른 축(환경·자율성). 식품 타깃 2개 동시 = 금지("한 번에 하나" 위반·반복 노출 흩어짐 → 옛 '휙휙' 회귀). 상한 1.5 · 하한 1.
미션 수명미션 ≠ 한 주에 끝내는 과제. "이번 주에 이 음식에 2~3회 적립" → 같은 미션을 2~3주+ 지속(다주 아크). '주'는 미션 수명이 아니라 재진단 주기.
'완료' 정의이벤트가 아니라 누적 상태. 1~2회 먹음 ≠ 완료(변동성 범위 안·내리면 수용 붕괴). 판정 = distinct dish 2+(마스터리 배지 기준) 또는 연속 2~3회 거부 없는 수용. 판정은 일요일 회고 권한, 평일은 미션 못 끔.
주중 조기 수용미션 종료 ❌ → 유지(공고화) 모드로 격하: 그날 축하(행동 0) → 남은 슬롯 push 끄고(채근 예산 선소진) 변주·칭찬으로 노출만 유지. 새 하드 미션 끼워넣기 = 안티패턴(축하 직후 새 숙제는 강화 파괴). 부모에겐 "정복/끝" 금지 → "친해지기 시작".

채근(주 1회) ≠ 노출(주 2~3회). 두 축을 혼동해 '주 1회만 노출'로 줄이면 노출 예산 부족으로 오히려 너무 느려진다. 노출(슬롯 배치)은 2~3회, 적극적 들이밀기(push 무브 mix/beside)만 1회.

4. 주간 ↔ 일간 관계 · 일관성 유지

핵심 질문: '일일이 주계획을 어떻게 닻으로 삼고, 어떻게 못 벗어나게 하나.' 3겹 가드로 잠근다.

가드규칙효과
① 슬롯 우선월~토는 selectScenario를 호출하지 않는다(LLM도 안 봄). 시나리오·타깃은 슬롯이 결정.'휙휙' 원천 차단. 주간 분배가 곧 일일.
② 화이트리스트
override만		그날 교정은 딱 3종만 허용: ⓐ 안전 인터럽트(전환→축하·적신호→관찰) ⓑ 타깃 소멸(슬롯 타깃이 그날 결핍 아니면 같은 미션군 내 다음 후보로만) ⓒ 무브 회전(시그니처 겹치면 표현만). 그 외 임의 변경 금지.닻에서 못 벗어남. 미션군 밖 이동 불가.
③ drift 감지실행 후 plan.frame이 슬롯과 다르면(안전 인터럽트 제외) context.driftFromPlan=true 기록.어드민에서 닻 이탈 모니터.

충돌 시 우선순위

dormancy(휴면 복귀) > shock(데이터 쇼크) > ARFID 게이트 > 닻 슬롯 준수 > planFor 자율(폴백)

닻 슬롯 채택 조건(AND): ⓐ slot.scenarioId가 13개 중 하나 ⓑ 그 시나리오 trigger(오늘 신호) 통과 ⓒ slot.target이 오늘 실신호에 존재 ⓓ shock/dormancy/등원변경 아님. 하나라도 실패 → 그 슬롯만 폐기, planFor 자율. 이 trigger 게이트가 LLM이 '없는 결핍을 미션으로 박는' 환각의 최종 방어선이다.

닻 전체 재작성은 오직 다음 일요일. 월~토는 슬롯을 그날만 덮을 수 있을 뿐, weekly_plans 자체는 주 내내 불변(안정감). 예외 1: 주중 결핍 구조가 근본 변동(red 영양소 2개+ 신규 추가 && 슬롯들이 하나도 안 다룸)이면 남은 평일 슬롯만 갱신(과거 슬롯·ledger 보존).

5. ⭐ 채근 정책 변경 — 주 0회 → 주 1회

이사님 지시: "채근도 이제는 아예 안 하기보다 주당 1회 정도는 하는 걸로 하자." 이 한 줄이 옛 plan 엔진의 핵심 가드를 뒤집는다.

옛 규칙 (plan 엔진)

  • planFor: 어제와 같은 타깃이면 무조건 다른 각도(정체기)로 전환
  • = 같은 걸 이틀 연속 미는 채근 0회
  • 부작용: 반복 노출이 구조적으로 불가능(같은 음식 주 2~3회를 못 함)

새 규칙 (주간 닻)

  • 주간 미션 타깃을 슬롯에 2~3회 의도 배치 = 반복 노출 = 정상
  • '직접 밀기(push)'는 ledger.pushUsed로 정확히 주 1회
  • 나머지 날은 같은 타깃이라도 부드러운 변주(조리법·궁합·환경·칭찬)

'채근(push)'의 정의 — 코드 판정

채근 = 같은 타깃을 적극적으로 들이미는 행동. 구체적으로 moveKey ∈ {mix, beside}('좋아하는 음식에 섞기'·'옆에 곁들이기')로 같은 타깃을 한 번 더 제안하는 것. 나머지 무브(recook·pair·cook-together·model·sensory)와 환경/자율성/칭찬은 변주이지 채근이 아니다.

주 1회 강제 — 결정론 카운트

장치규칙
상한(주 1회)일요일 분배가 슬롯 중 정확히 1개에만 push:true 마킹(미션 핵심 타깃·주 중반 화/수). 실행일에 pushUsed=false면 적용 + pushUsed=true·pushDay=today 기록. 이미 true면 강등(moveKey를 push계열→비push계열로, 타깃은 유지=반복 노출 지속·들이밀기만 제거). LLM이 욕심내 2개 박아도 코드가 1회로 클램프.
하한(0 허용)push 슬롯이 안전 인터럽트에 먹히면 같은 주 다음 평일로 1회 재부착(쿨다운 1일). 주말까지 못 밀면 그 주 0회 — '최대 1회'이지 '의무 1회'가 아니다.
0회 강제잠금ARFID 의심(icfqRiskCount≥2·증상 메모·먹는 종류 감소) 또는 휴면 복귀 주는 종합 시점에 pushUsed=true 선잠금 → 그 주 채근 0회. 압박이 이탈·역효과인 아이를 보호.

왜 DB 컬럼 하나로 충분한가: 채근 카운트가 weekly_plans.ledger.pushUsed 한 곳에만 있어(단일 진실), 일일 재실행(force/QA)에도 멱등이고 별도 카운터가 필요 없다. '주 1회'가 데이터 구조에 박혀 자연히 강제된다.

시그니처 dedup은 유지. 같은 타깃이 여러 날 와도 (frame·move·도입)이 최근 3일과 겹치면 무브·도입을 회전 → 복붙 편지 방지. dedup의 차단 대상이 '의미(같은 처방)'에서 '표현(같은 문장틀)'으로 좁아진다(타깃 반복은 주간 미션이 책임지므로 허용).

8. 적대 엣지케이스 10종 + 결정론 안전망

2층을 깨려고 시도한 10개 케이스. 안전 제1원칙 = 일간 결정론은 닻 유무와 무관하게 항상 단독으로 편지를 낼 수 있어야 한다(닻은 '있으면 좋은' 안정 장치). 모든 완화책은 weekly_plans 1개 테이블 + planFor 폴백에 수렴한다.

#케이스결정론 안전망
①신규 자녀(일요일에 7일 이력 없음)LLM 미호출. cold_synth — 결정론 온보딩 4주 아크 슬롯(관찰·신뢰, 거부 단정 금지). mission=null. 비용 0.
②휴면 → 복귀(주중)휴면 경로(reengageBlock)가 닻보다 우선. 복귀일 shockResetAt 기록 → 남은 슬롯 준수 강제 안 함. push 선잠금(복귀 직후 채근=이탈 위험).
③저데이터 주(기록 듬성)슬롯 trigger가 통과 못 함 → planFor 자율. 주간은 degraded, 미션=행동 금지·"기록 채우기 한 줄". LLM이 미션 만들어도 화이트리스트 교집합 비어 target 전부 null 강등 → plateau 수렴.
④주중 데이터 쇼크(새 거부 폭증·전환)쇼크 게이트(신규 거부 ≥3 OR 전환 발화 OR icfqRisk 점프) → 닻 슬롯 폐기·그날 신호로 자율. progress-celebrate(100)/neophobia(92)가 어차피 닻을 이김.
⑤주계획 진부화(부모 미션 무시)⭐ 진부화 타이머(부모 무시)와 진척 타이머(수용 누적 중)를 분리. missionTouch 있고 수용 오르는 중 = 3주 캡 해제·같은 미션 지속(8~10회 도달 직전 끊기 방지). 부모가 2주째 손도 안 댄 미션만 같은 타깃 반복 금지·각도 전환(섞기 실패→함께 요리).
⑥다자녀weekly_plans는 child_id PK라 자녀별 독립 닻. 알림은 현행 notifiedParents로 부모당 1건. 미션은 자녀 탭별 표시.
⑦일요일 종합 LLM 실패/비용try/catch → 지난주 닻 있으면 carry(stale_carried, 안정감>최신성). 없으면 cold_synth. basis_hash 동일 시 LLM 스킵. 일간은 절대 영향 없음.
⑧기관 변경·등원토글basis_attends_daycare ≠ 현재면 집-기관 관련 슬롯(home-daycare-gap·nutrient-gap) 무효화·해당 슬롯만 자율. 등원 ON 직후 첫 주는 re-exposure 슬롯 보류(신호 미성숙).
⑨주 경계(부모가 주중 가입)그 주 닻 없음 = planFor 폴백(정상·무편지 0). 첫 일요일에 7일 이력으로 정식 점화(이력<3이면 cold_synth). "가입 주는 닻 없이 일간, 첫 일요일에 닻 점화."
⑩채근 1회가 역효과인 아이(ARFID)icfqRiskCount≥2 등 ARFID 신호 → 종합 시점에 pushUsed=true 선잠금(채근 0회). neophobia는 PRAISE_ONLY라 target=null = 밀 게 없음. 2층이 채근을 못 켠다.

결정론 안전망 요약: ① 닻 부재·LLM 실패 → planFor 폴백(무편지 0) ② LLM 환각 target → 화이트리스트 교집합 강등 ③ 환각 scenarioId → 13개 검증 실패 시 드롭 ④ ARFID·휴면 → push 선잠금 ⑤ 쇼크 → 닻 무시·자율 ⑥ 신규/저데이터 → cold_synth·degraded(LLM 미호출) ⑦ push 상한 → 코드 클램프 1회.

9. 결정 필요 — 이사님 확인

Q1. 주 경계. isoWeekKey는 월요일 시작(일요일=전주 끝). '일요일 종합 → 당일부터 다음 주(월~토) 적용'이 자연스러우려면 week_key를 다음 주 키로 +1 보정해야 함(아니면 월요일 첫 편지가 지난주 닻을 읽음). → 권장: 일요일 종합은 다음 주 키로 저장.
Q2. 일요일 부하. 자녀당 종합 LLM +1콜이 일요일에 몰림(현재 활성 ~4명은 무해, 키트 캠페인 유입 후 재검토). 못 끝낸 자녀는 월요일 lazy 폴백(품질 동일, 분배일만 다름). → 권장: 허용, 스케일 시 self-fetch 배치 체이닝.
Q3. 미션을 부모에게 노출? "이번 주 미션" 카드를 명시하면 진부화(무시)가 가시화되지만 미션 못 지킨 죄책감 위험(압박 역효과=mealtime-atmosphere가 경계). → 권장: 편지 안에 녹이기(현행 톤), 별도 카드는 보류.
Q4. 채근 톤. '주 1회 같은 타깃 한 번 더 밀기'가 무압력 원칙(Satter 식사 분담)과 충돌 안 하게 promptHint 문구를 짜야 함. → 권장: 적대검수 워크플로 1회(채근이 잔소리로 읽히면 역효과).
Q5. 구조화 입력 신뢰도. autonomy/environment는 부모 입력 빈도가 낮을 수 있음(care 칩 optional). 임계 미달 시 환경 테마를 안 띄우는 보수 정책 vs '입력 권유' 유도(profileNudge처럼)? → 권장: 보수(데이터 부족 단정 금지·P4) + 가벼운 입력 권유.
