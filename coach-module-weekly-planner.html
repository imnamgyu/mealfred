📅 이 모듈은 '작전 사령부'입니다. 일요일 종합(① 신호선택기의 runWeeklyPlanning·Sonnet)이 잡은 빈약한 닻(mission_target=식품군 카테고리 1개) 위에, 결정론 후처리 enrichWeeklyPlan(synth, ctx)이 다른 모든 모듈을 호출(오케스트레이션)해 한 주치 구체 계획(plan_detail)을 미리 굽습니다 — LLM 0콜·순수함수·throw 시 null degrade. 추천엔진(②)의 popularDishesFor로 카테고리를 실제 음식으로 펼치고, 네트워크(⑦)의 verifiedCousinsOf로 도전 사촌을 붙이고, 영양평가(④)의 bmiBand/growthTracking으로 탄단지 macro 트랙을 깔고, 진척(⑤)의 goals로 anti-stall을 가시화합니다. 월~토 일간 전술층(① 신호선택기)은 pickPlanSlot으로 그날 데이터에 맞춰 슬롯을 소비합니다. 신규 영속 필드: weekly_plans.plan_detail jsonb(additive·degrade-safe).

비개발자용 — 이렇게 운영됩니다

이 모듈은 '한 주 식단표를 미리 짜는 주방장'입니다. 식당에 비유하면, 일요일 밤에 주방장이 다음 한 주(월~토) 무엇을 어떤 순서로 낼지 메뉴판을 통째로 미리 짜두는 일입니다. 왜 필요했냐면, 예전 주간계획은 너무 빈약했습니다. 한 주의 목표가 '콩류'라는 식품군 카테고리 딱 하나였고, 후보도 카테고리 4개 이하였습니다. 그러니 월요일도 콩류 → 두부, 화요일도 콩류 → 두부, … 몇 주 내내 '콩류 도돌이표'가 됐습니다. 이사님 지적의 핵심이 이것입니다.

그래서 이 모듈은 주간계획을 진짜 작전 사령부로 격상했습니다. 일요일 종합이 '이번 주는 콩류'라고 큰 방향만 잡으면, 그 직후 이 모듈이 다른 모든 엔진을 한꺼번에 불러 7일치 구체적인 음식 회전표를 미리 굽습니다. '콩류'를 두부조림 → 순두부찌개 → (사촌 도전) 검은콩 → … 처럼 실제 음식 이름 7칸으로 펼쳐, 같은 식재료가 이틀 연속 나오지 않게 짭니다. 두 갈래를 번갈아 섞습니다 — '부족한 걸 보충하는 칸(supply)' 둘에, '잘 먹는 음식의 검증된 사촌으로 도전하는 칸(challenge)' 하나. 이 도전 칸이 '콩류 3주 도돌이표'를 구조적으로 끊습니다.

여기에 세 가지를 더 깝니다. ① 키·몸무게(BMI/성장곡선)를 봐서, 또래보다 가볍거나 성장이 더딘 아이면 '탄단지(단백·지방·열량) 보강 트랙'을 켜고 회전표에 고기류를 앞에 끼워넣습니다(격주). ② 부모 교육 '수업'이 3주째 제자리면 그게 보이게 표시해, 일간 편지가 '지난주 X에서 이번주 Y로'라고 진도를 말할 수 있게 합니다. ③ '오늘의 영양 거울 한 줄'도 7일치로 미리 짭니다 — 결핍이 콩류 하나뿐이면 매일 같은 콩류 클로징이 앵무새가 되니 격일로만 넣습니다. 마지막으로, 이 표는 어디까지나 가이드입니다. 월~토 그날 아침에 일간 엔진이 그날 기록 데이터로 다시 검수해서, 이미 채워진 칸은 건너뛰고 다음 유효한 칸으로 적응합니다.

🥗 영양·BMI는 이렇게 처리합니다
탄단지(macro) 트랙은 추측이 아니라 우리가 계산한 성장 수치만 씁니다. 영양평가(④)가 준 BMI 밴드(bmiBand)와 성장곡선 추종도(growthTracking)를 봅니다. 저체중이거나 성장곡선이 자기 채널에서 아래로 이탈(주의·경고)이면 '탄단지 보강군'(고기·계란·콩류·유제품)을 켜서 supply 회전표 앞에 고기류를 주입하고, 부모 편지엔 growthTrackToPhrase가 만든 P10 톤 한 구절(숫자·체중·살·BMI 단어 0)을 넣습니다 — 이게 이사님이 말한 '키·몸무게 고려해서 탄단지를 높이는 트랙, 타깃=고기류'입니다. 과체중·비만이면 거꾸로 간식 절제(snackRestraint)로 안내합니다. 정상이면 트랙을 끕니다(불필요한 잔소리 0). 그것도 격주로만 노출합니다 — 가정 측정은 노이즈가 크고 자주 말하면 부담이 되니, arcWeek이 짝수인 주에만 macro 줄을 넣습니다.
🧒 운영 예시(아린·5세) — 아린의 일요일 종합이 잡은 닻: 주력 타깃=콩류(집에서 단일 결핍), 비타민A채소도 약함, 잘 먹는 음식=소고기·계란·감자·미역, BMI 밴드=정상. 예전이라면 월~토 내내 '콩류 → 두부'였습니다. 이 모듈은 그 직후 7칸 회전표를 굽습니다: ①두부(콩류·supply, 두부조림·순두부찌개) → ②단호박(비타민A채소·supply, 단호박죽) → ③검은콩(콩류·challenge·소고기의 검증 사촌, 콩조림) → ④콩나물(콩류·supply, 콩나물무침) → ⑤시금치(비타민A채소·supply) → ⑥두부(두 번째 dish로) → ⑦… 같은 식재료가 이틀 연속 없이 7칸이 채워집니다. 영양 거울도 콩류가 단일 결핍이면 격일로만('K-04b 쿨다운'). 만약 아린이 저체중이었다면, 회전표 맨 앞에 소고기·달걀·치즈가 끼워지고('고기류 타깃'), 격주로 "또래보다 가벼운 편이라 끼니 양과 단백질·열량을 조금 더 챙겨주면 좋아요" 한 구절이 들어갑니다. 월요일 아침, 일간 엔진은 슬롯 인덱스 (daySeed+cidHash)%7로 한 칸을 고르되, 그 칸이 supply인데 그날 보니 콩류가 이미 채워졌다면 다음 유효 칸으로 회전합니다(유연성 가드).
🔧

증상 개선 — 구현 · 기능 · 설계 변경 방향 (2026-06-18 이사님 지시)

★최상위 근본원인 — 주간계획이 '얇았다'. 닻의 mission_target이 식품군 카테고리 1개('콩류')였고 target_pool도 카테고리 ≤4개여서(coachWeekly.ts resolveGoals/coldSynth), 월~토 일간 편지가 매번 같은 카테고리의 첫 대표(GROUP_INGREDIENTS['콩류'][0]=두부)로 붕괴했다. 결과로 '콩류 → 두부' 도돌이표가 몇 주씩 갔다. 이 모듈은 주간계획을 일급 오케스트레이터로 격상해, 일요일 종합 직후 결정론 후처리 enrichWeeklyPlan이 다른 모든 모듈을 호출해 카테고리가 아니라 구체 dish 7칸 회전·2트랙·macro·거울 스케줄을 미리 굽고, 일간은 pickPlanSlot으로 그날 데이터에 맞춰 소비한다.

3주째 동일 수업 — 커리큘럼 정체가 일간에 안 드러남focus/standby·피벗 가시화 부재 — DESIGN_GAPenrichWeeklyPlan의 curriculum 블록이 focusUnit·standby·stalledOut·graduatedTo·lever를 산출(applyFocusFatigue를 이미 거친 goals에서). graduatedTo=직전 주 focus와 다르면 '졸업'으로 표시 → 일간 편지가 '지난 X→이번 Y' 진도를 말할 수 있다. 진척 영속(advanceProgress)은 ⑤ 진척·커리큘럼 소관, 본 모듈은 '주간 스냅샷 가시화'만.P1

일간이 그날 데이터를 무시 — 주간 가이드를 맹목 추종하면 이미 채운 결핍을 또 민다주간계획이 '명령'이면 적응 불가 — DESIGN_GAP⭐유연성 가드(이사님): 주간계획=후보 가이드, 일간=그날 데이터로 종합결정. pickPlanSlot이 supply 슬롯의 group이 deficitNow에 더는 없으면(받아들임/채워짐) 다음 유효 슬롯으로 회전. challenge 슬롯은 항상 유효(도전이라 결핍 무관). 일간 두뇌 + 안전 인터럽트(전환 축하·ARFID)는 이 위에서 여전히 override.P2
📅 이 모듈이 근본적으로 바꾸는 것 — 주간계획은 이제 '카테고리 1개'가 아니라 '7일치 구체 계획 한 묶음(plan_detail)'을 미리 굽는 작전층이다. 핵심은 셋. ① 일요일에 다른 모듈을 전부 호출(오케스트레이션)해 카테고리를 실제 dish로 펼치고(②), 도전 사촌을 붙이고(⑦), 탄단지 macro를 깔고(④), anti-stall을 가시화한다(⑤) — 단 LLM 0콜·순수·throw 시 null degrade. ② supply 2 : challenge 1 인터리브 + 연속 동일 식재료 금지로 도돌이표를 구조적으로 끊는다. ③ 일간은 pickPlanSlot으로 그날 결핍 데이터에 맞춰 슬롯을 vetting한다 — 주간은 가이드, 일간은 적응. plan_detail은 additive 캐시(기존 컬럼이 SSOT), null이면 일간이 옛 buildIngredientPool 회전으로 자동 폴백한다.
🧒 한 줄 쉬운말 — 예전엔 '이번 주는 콩류' 한 마디만 정해놓으니 매일 두부만 나왔습니다. 이제는 일요일 밤에 한 주치 음식표(두부조림→단호박→검은콩→…)를 미리 통째로 짜두고, 키·몸무게에 따라 고기류를 더하고, 매일은 그날 상태를 보고 그 표에서 한 칸씩 골라 씁니다.

제한·잔소리의 역효과(Satter)·무압력정상 밴드면 macro 트랙을 끄고(잔소리 0), macro도 격주로만 노출. 과체중도 '금지'가 아니라 간식 '교체'(단 음료→과일·요거트)로 안내. 끼니 다양성은 유지
주제 피로 캡(focus fatigue)커리큘럼 가시화에 applyFocusFatigue를 거친 goals를 쓴다 — 같은 유닛 2주 연속 focus이고 step 전진 0이면 standby로 강등돼, graduatedTo로 '수업 전환'이 일간에 드러남
괴식 차단(검증된 조합만)모든 dish는 popularDishesFor(매운·튀김 필터·또래 인기 빈도)로만, 사촌은 verifiedCousinsOf(verified만), 궁합은 strongPairsOf(강신호만)로만 — 지어낸 조합 0(괴식 차단)

요약

라이브📅 작전층 오케스트레이터로흡수: coachWeekly.ts enrichWeeklyPlan + pickPlanSlot

빈약한 '카테고리 1개' 닻이 일간 도돌이표의 근원이었다(이사님 2026-06-18). 이 모듈은 일요일 종합(Sonnet runWeeklyPlanning) 직후 결정론 후처리 enrichWeeklyPlan(synth, ctx)이 다른 모든 모듈(추천엔진·네트워크·영양평가·진척)을 오케스트레이션해 7일치 구체 계획(plan_detail)을 미리 굽는다 — LLM 0콜·순수·throw 시 null degrade. 카테고리를 실제 dish 회전으로 펼치고(supply 2 : challenge 1·연속 동일 금지·도돌이표 구조적 차단), BMI/성장으로 탄단지 macro를 깔고, 진척으로 anti-stall을 가시화하고, 영양 거울을 7일치 쿨다운 스케줄로 짠다. 월~토 일간 전술층(① 신호선택기)은 pickPlanSlot으로 그날 결핍 데이터에 맞춰 슬롯을 vetting해 소비한다(주간=가이드·일간=적응). 신규 영속 필드 weekly_plans.plan_detail jsonb(additive·SSOT=기존 컬럼·degrade-safe).

▼

2) macro 트랙 먼저 계산
buildMacroTrack(ctx) — bmiBand+growthTracking으로 저체중/성장더딤이면 boostGroups=['고기·계란','콩류','유제품'](탄단지) 활성·growthTrackToPhrase P10 구절, 과체중이면 snackRestraint, 정상이면 비활성. cadenceWeek=arcWeek%2===0(격주). ⭐먼저 계산 — supply 풀에 고기류를 앞에 주입하려면 macro 결과가 필요

pickPlanSlot(detail, { daySeed: 12, cidHash: 5, deficitNow: new Set(["콩류","비타민A채소"]) })
// → base=(12+5)%5=2 → slot[2]=검은콩(challenge·항상 유효)
//   { slot: {ingredient:"검은콩", dishes:["콩조림"], track:"challenge"...},
//     slotIndex: 2, mirror: {line:"어린이집 덕에…", kind:"deficit"}, macroPhrase: null }
// → recoIng="검은콩"·recoMode="challenge" → 작문기(⑥)에 dish·mirror 전달
📤 전달 — PlanDetail → weekly_plans.plan_detail(영속) → 월~토 pickPlanSlot이 그날 슬롯을 픽 → cron이 recoIng(=slot.ingredient)·recoMode(track)·mirrorLine(slot.mirror.line)·growthMirror(planSlotCtx.macroPhrase ?? growthMirrorCtx)를 작문기 LetterInput에 주입. 어드민 감사용으로 context.planSlot(slotIndex·dishes·track·mirror·macro)을 적재. ⭐유연성: supply 슬롯의 group이 그날 결핍이 아니면(deficitNow에 없으면) 다음 유효 슬롯으로 회전.
4

CoachContext 전달 — 📅 작전층 부가 계획으로

이 모듈이 채우는 CoachContext/닻 필드(통합 아키텍처 §3 데이터 계약):

plan_detailplanSlotrecoIngrecoModemirrorLinegrowthMirror(macroPhrase)

일요일 enrichWeeklyPlan이 weekly_plans.plan_detail(jsonb)을 채우고, 월~토 pickPlanSlot이 그날 소비한 슬롯을 context.planSlot(slotIndex·ingredient·dishes·group·track·via·mirror·mirrorKind·macro)으로 적재한다. 작문기(⑥)에는 recoIng(슬롯 식재료)·recoMode(supply/challenge)·mirrorLine(거울 줄)·growthMirror(macroPhrase 우선)가 전달된다. plan_detail이 null이면 이 필드들은 옛 buildIngredientPool 회전 값으로 채워진다(degrade).

5

가드 — 괴식·환각·안전 차단

  • 괴식 0 — 모든 dish는 popularDishesFor(매운·튀김 필터·또래 인기 빈도)로만, 사촌은 verifiedCousinsOf(verified만), 궁합은 strongPairsOf(strong 강신호만)·PLAN_MEAL_GROUPS 식품군 한정 → 지어낸 조합·약신호 곁들임 불가
  • degrade-safe — enrichWeeklyPlan은 throw 가능하나 호출자(cron)가 try/catch로 null degrade. plan_detail 컬럼 미적용 시 legacy upsert로 plan_detail만 빼고 닻 영속 보장. plan_detail null → 일간이 옛 buildIngredientPool 회전으로 폴백(닻 유무와 무관하게 발행)
  • SSOT 보존 — plan_detail은 additive 캐시일 뿐, 기존 컬럼(mission_target·target_pool·budget·ledger·goals)이 단일 진실. 일간 잠금·채근 캡·아크는 기존 컬럼이 계속 좌우(plan_detail은 'dish 회전·거울·macro'만 부가)
  • 유연성 vetting — 주간계획=후보 가이드, 일간=그날 데이터로 종합결정. pickPlanSlot이 supply 슬롯 group을 deficitNow로 검수해 이미 채운 결핍 재추천을 차단. challenge는 도전이라 결핍 무관 항상 유효. 일간 두뇌·SAFE_INTERRUPT(전환 축하·ARFID)는 이 위에서 override
  • macro 격주 — 탄단지 트랙은 cadenceWeek=arcWeek%2===0(격주)일 때만 노출, 2주 연속 금지(측정 저빈도·부담 존중). 정상 밴드면 active:false로 트랙 자체를 끔(불필요한 잔소리 0)
  • 거울 쿨다운(K-04b) — 단일 결핍은 격일(짝수 슬롯)만 결핍줄, 홀수는 칭찬/생략 → 매일 같은 콩류 클로징 앵무새 차단. 다중 결핍은 라운드로빈으로 같은 군 연속 금지
  • 연속 동일 식재료 금지 — 7-슬롯 회전이 pick.ingredient===lastIng이면 skip → 같은 식재료 이틀 연속 불가('콩류 3주 도돌이표' 구조적 차단)
  • QA 시뮬 안전 — 일요일 synth는 ?date= QA 오버라이드에선 금지(과거 일요일 재실행이 라이브 닻·ledger를 DEFAULT로 파괴하던 버그 봉합)
