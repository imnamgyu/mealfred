비개발자용 — 이렇게 운영됩니다

이 모듈은 코칭 엔진이 "이 식재료, 어디에 넣으면 아이가 잘 먹을까?"를 물었을 때 대답해 주는 '음식 백과사전 검색대'입니다. 직접 결정을 내리지는 않고, 다른 코칭 모듈이 추천 편지를 쓸 때 필요한 사실 4가지를 빠르게 찾아 줍니다. ① 궁합(같이 차리면 자연스러운 짝, 예: 두부+다시마) ② 사촌(맛·식감이 닮아 갈아탈 만한 대체 음식, 예: 두부→콩나물) ③ 음식 적합도(이 식재료를 넣기 좋은 요리, 예: 시금치→무침·나물) ④ 급식 빈도(이 식재료가 또래 급식에 얼마나 자주 나오나, 예: 당근은 상위 2%로 아주 흔함). 이 네 가지는 각각 우리가 모은 실제 레시피·급식 데이터에서 미리 계산해 둔 표를 그냥 '읽어 오기'만 하는 역할입니다.

왜 필요하냐면, 코칭의 핵심 전략이 "잘 먹는 음식에 결핍 식재료를 슬쩍 섞기"인데, 아무 조합이나 섞으면 '미역국에 당근'처럼 아이가 질색하는 괴식이 나옵니다. 그래서 추천하기 전에 반드시 이 검색대에서 "이 짝이 실제 레시피에 충분히 같이 나왔나? 닮은 사촌이 검증됐나? 이 요리에 넣어도 자연스럽나?"를 확인합니다. 즉 코치가 상상으로 조합을 지어내지 못하게 막아 주는 '현실 검문소'입니다.

기술적으로는 51KB짜리 큰 데이터를 아이 휴대폰 앱에 통째로 싣지 않으려고, 빌드(앱을 만드는 시점)에서만 필요한 부분을 뽑아 쓰도록 짜여 있습니다. 또 데이터 출처를 graphSource 한 곳으로 모아서, 나중에 데이터를 파일(JSON)에서 데이터베이스(SQL)로 바꿔도 추천 로직은 한 글자도 안 건드리고 갈아끼울 수 있게 설계돼 있습니다.

🥗 영양·편식 정보는 이렇게 처리합니다
이 모듈이 실제로 적용하는 편식·괴식 안전장치는 세 가지입니다. ① 약한 궁합 차단: pair(궁합)는 강도 2 이상(또는 등급 'strong')만 추천에 씁니다. '떡+달걀'(lift 0.72, 강도 1) 같은 약한 짝이 괴식 사고를 낸 적이 있어, strongPairsOf가 약신호 조합을 일괄로 걸러 냅니다. 이 임계값(2)은 음식×식재료 매트릭스의 기준과 통일돼 있어, 추천을 쓰는 모든 모듈(coachRecos·coachMaterials·comboMatrix·comboGuard)이 같은 문을 통과합니다. ② 검증된 사촌만 갈아타기: 푸드체이닝(닮은 음식으로 점프해 가며 새 음식 노출)에 쓰는 bridge(사촌)는 verified(검증됨) 표시가 붙은 것만 씁니다. 두부→콩(대두)·콩나물처럼 사람이 큐레이션한 고신뢰 사촌만 통과합니다. ③ '결핍을 0회인데 흔하다'로 위장 금지: 급식 빈도는 실제로 나온 횟수(freq>0)만 유효로 보고, 단호박·요거트처럼 급식에 0회 등장한 식재료는 '상위 100%'로 둔갑시키지 않고 정직하게 null(모름)로 둡니다. 즉 데이터가 없으면 없다고 말합니다. 이 모듈 자체는 결핍 판정이나 강요 금지 같은 '판단'은 하지 않습니다 — 그건 상위 코칭 모듈의 몫이고, 여기서는 그 판단에 필요한 '사실'만 안전하게 공급합니다.
🧒 운영 예시(아린·5세) — 아린(5세, 콩류·채소 결핍, 소고기·계란은 잘 먹고 미역국 좋아함)에게 콩류를 늘려 주고 싶다고 합시다. 입력: 코칭 엔진이 "두부(콩류)를 어디에 어떻게 곁들일까?"를 이 검색대에 물어봅니다. 처리: ① 궁합 조회 → 두부의 강한 짝으로 다시마(94회)·계란(43회)·멸치(111회)·무(70회)가 나옵니다(약한 '두부-소고기' lift 0.38은 차단). 아린이 계란을 잘 먹으니 '두부+계란'이 안전한 곁들임으로 뽑힙니다. ② 음식 적합도 조회 → 두부는 순두부(3점)·계란찜(3점)·국(3점)에 자연스럽습니다. ③ 사촌 조회 → 두부의 검증된 사촌은 콩나물·콩(대두)이라, 나중에 두부에서 콩나물로 한 걸음 갈아탈 길도 확보. ④ 급식빈도 → 콩류는 결핍이고 아린이 좋아하는 당근은 이미 상위 2%로 충분하니 콩류 쪽에 우선순위. 결과: 엔진은 "잘 먹는 계란에 두부를 섞은 계란찜" 또는 "아린이 좋아하는 미역국 결에 닿는 순두부"를 근거 있는 인기 음식 이름으로 추천하고, '미역국+당근' 같은 괴식이나 약한 조합은 검문소에서 걸러집니다.

문헌 · 이론 근거

이 모듈이 실제로 구현·적용하는 이론입니다(코드의 근거 기준).

이론 · 문헌어떻게 적용하나
레시피 공출현 lift (장바구니 분석/연관규칙)우리 레시피 코퍼스에서 두 식재료가 우연 이상으로 같이 등장한 정도(lift)와 동시출현 횟수(count)로 궁합 강도를 매기고, 약한 짝은 추천에서 차단한다.
Food chaining (푸드체이닝, 점진적 음식 사슬 노출법)이미 잘 먹는 음식에서 맛·식감이 닮은 '사촌'으로 한 걸음씩 갈아타며 새 음식을 노출하는 방식을, 검증된 bridge 엣지로 구현한다.
NEIS 식판 단위 공통출현 (같은 끼니 동시 차림)전국 초등 급식 식판에서 함께 차려진 식재료를 tray 등급으로 그래프에 추가해, 레시피엔 없어도 '같은 끼니에 자연스럽게 차려지는' 짝을 보강한다.
농진청 영양성분 맵 (nutrient-map)식재료별 주요 영양소를 농촌진흥청 표준 데이터로 매핑해 graphSource가 같은 출처에서 함께 제공한다(결핍 해석의 근거).

엔진 소비·CoachContext 주입
급식빈도는 recoEvidence(topPctOf)·coachMaterials 랭킹, 궁합/적합도는 comboMatrix·comboGuard 괴식검증으로 흐르고, 최종 text는 cron이 bridgeFacts로 CoachContext에 주입(두뇌 useFood=false면 빈 문자열).
2

핵심 함수

함수역할
neighborsOf(nm)한 식재료의 모든 이웃(궁합 pair·사촌 bridge)을 bridge먼저→grade→strength→count 순으로 정렬 반환. affinity 정적조회 시드.
strongPairsOf(nm)곁들임 추천용 강한 궁합만 — grade='strong'(없으면 strength≥2). 약신호 곁들임(떡+달걀) 일괄 차단. 전 소비자(coachRecos·coachMaterials·comboMatrix·comboGuard) 일원화.
verifiedCousinsOf(nm)푸드체이닝 chain 추천용 검증된 사촌(bridge·verified)만. 미검증 사촌으로 인한 환각 차단.
dishesForIngredient(nm, minScore=2)이 식재료를 넣기 좋은 음식(LLM 적합도≥minScore, 점수순). 도감 상세·코칭 인기음식·comboGuard 점수조회.
freqOf/topPctOf/isCommon(nm)식재료 급식빈도 메타(freq·rank·topPct). 0회(단호박·요거트)는 정직하게 null(상위100% 위장 금지). 4기준 랭킹·근거 문구 단일 진실원.
warmGraphFromSql(db)SQL→이름shape 캐시 교체(L1 warm). 안전게이트·고아엣지 드롭·dishes 메타는 JSON 유지·수치만 교체. 실패 시 JSON degrade.
getEdges/getDishMatrix/getRecipeFreq/getIngredientsLightgraphSource 단일 읽기경로 게터(동기·lazy). 엔진 모듈의 JSON 직접 import 금지 → 전환 시 누락 방지.
3

입력 / 출력 예시

📥 잘 먹는 식재료 목록 + 오늘 타깃 식품군(콩류)·일일 회전된 타깃 식재료(검은콩) + 또래 급식빈도맵. 단위 헬퍼는 식재료 이름 1개를 받는다.

📥 INPUT
// coachRecos.buildRecoFacts (네트워크 헬퍼를 묶어 호출하는 진입점) — 아린(5세) 크론 입력
{
  "likedIngredients": ["소고기", "계란", "감자", "미역", "두부", "쌀"],
  "target": "콩류",
  "targetIngredient": "검은콩",
  "freqMap": {
    "두부": 92,
    "검은콩": 14,
    "고등어": 60
  }
}

// 내부에서 호출되는 단위 헬퍼 예:
// strongPairsOf("검은콩")  → 검은콩의 강한 궁합(grade=strong / strength≥2)
// verifiedCousinsOf("미역") → 미역의 검증된 사촌(bridge·verified)
// dishesForIngredient("검은콩", 2) → 검은콩 넣기 좋은 음식(LLM 점수≥2)
// topPctOf("당근") → 당근 급식 상위 백분위
📤 OUTPUT
// buildRecoFacts 반환 (RecoFacts) — bridgeFacts로 CoachContext에 들어감
{
  "target": "콩류",
  "cousins": ["병아리콩"],
  "lines": [
    "[오늘 타깃 콩류] 삶은 검은콩 (또래 인기 음식: 콩자반·검은콩밥 · 잘 먹는 두부 곁들이면 좋아요)",
    "미역 → 사촌 다시마(또래 인기: 다시마채조림) / 궁합 두부·들기름",
    "감자 → 궁합 계란·당근"
  ],
  "text": "[오늘 타깃 콩류] 삶은 검은콩 (또래 인기 음식: 콩자반·검은콩밥 · 잘 먹는 두부 곁들이면 좋아요)\n미역 → 사촌 다시마(또래 인기: 다시마채조림) / 궁합 두부·들기름\n감자 → 궁합 계란·당근"
}

// 단위 헬퍼 반환 shape 예:
// strongPairsOf("검은콩") → [{ nm:"두부", kind:"pair", strength:3, basis:"같이 쓰는 레시피 N개", grade:"strong" }]
// topPctOf("당근") → 2   (당근 freq184·상위2%)
📤 전달 — RecoFacts.text가 cron에서 bridgeFacts로 CoachContext에 주입(두뇌 useFood=false면 ''). cousins는 어드민·회전 이력, lines는 손(작문)이 목록 밖을 못 지어내게 하는 그라운딩 근거. freq/궁합/적합도는 coachMaterials 랭킹·comboGuard 괴식검증으로도 흐른다.

가드 — 괴식·환각·안전 차단

  • 약신호 곁들임 차단: strongPairsOf가 grade='strong'(없으면 strength≥PAIR_MIN_STRENGTH=2)만 통과 — 떡+달걀(lift0.72) 등 s=1 약신호 추천 경로 일괄 차단
  • 미검증 사촌 차단: verifiedCousinsOf가 verified만 — 환각 사촌 푸드체이닝 방지
  • 환각 차단: buildRecoFacts는 popularDishesFor·STAPLE_FORMS 테이블 근거만 직조('목록 밖 음식·조합 금지', 편지가 지어내지 못함)
  • 정직 null: freqOf/topPctOf가 0회 식재료(단호박·요거트)를 '상위100%'로 위장하지 않고 null 반환
  • 주식 곡물 형태: stapleDisplay/STAPLE_FORMS로 날 곡물명(쌀·밀) 대신 먹는 형태(밥·빵·떡)만 노출
  • SQL safe degrade: warmGraphFromSql 안전게이트(edges<100·cells<10)·고아엣지(id 미해소) 드롭, 실패 시 JSON 스냅샷 유지
  • 클라 번들 누수 방지: JSON은 게터 안에서만 참조(tree-shake 유지) — 클라엔 그래프 51KB 미적재
