비개발자용 — 편지 한 통이 만들어지는 과정

매일 새벽, 아이 한 명마다 아래 5단계가 돕니다. 코드를 몰라도 읽히게 정리했습니다.

① 오늘의 '아이 상태표' 만들기
어제까지의 집·기관 식단 기록을 모아 영양 신호등(부족/충분)·잘 먹는 음식·거부한 음식·새 음식 노출 진척을 한 장으로 계산합니다. 전부 실제 기록에서 나온 값이라 없는 사실을 지어내지 않습니다.
▼
② '뇌'가 오늘의 각도 + 이번 주 목표 고르기
13개 코칭 각도(예: 영양 공백·식사 분위기·진전 축하) 중 오늘 가장 도움될 1개를 고르고, 이번 주 가르칠 교육 목표(커리큘럼 유닛)도 정합니다. 매일 다른 각도로 돌려 잔소리 반복을 막습니다.
▼
③ '추천엔진'이 채울 음식 정하기
부족한 영양을 채울 실제로 잘 먹는 음식의 사촌·곁들임을 검증된 데이터에서만 뽑습니다(예: "미역국에 두부 조금"). 조리법을 함부로 지어내지 않아 괴식이 안 나옵니다.
▼
④ '손'이 따뜻한 편지로 쓰기
위 재료를 받아 따뜻한 한국어 편지로 옮깁니다. 강요·괴식·과한 소금은 안전가드가 거르고, 발행 전 한 번 더 검수합니다.
▼
⑤ 부모 실천 → 새 기록 → ①로
부모가 한 걸음 실천하고 기록하면, 그게 내일 상태표에 반영돼 코칭이 점점 그 아이에게 맞춰집니다.
두 가지 핵심 약속 — (가) '무엇을' 줄지는 코드가 결정론으로, '어떻게' 쓸지(문장)만 AI가 → 괴식·환각 0. (나) 판단은 하루 한 번 미리 계산해 저장(아이 현황 스토어), 편지는 그걸 읽어 글만 → 1만 명도 감당. 아래 §0~는 이 구조의 상세이고, 각 부품의 운영·문헌·테이블은 🧩 모듈 명세 7종에 있습니다.
0

네이밍 규칙 — 엔진 / 모듈 / 데이터 레이어

확정

세 가지로만 부른다. 기준은 "산출물과 루프를 가지면 엔진, 데이터를 돌려주면 모듈, 읽고 쓰는 공용 데이터면 레이어".

⚙️ 엔진 (Engine)
사용자 향 산출물을 만들고 자기 루프(크론)를 가진 오케스트레이터. 시스템에 하나.
코칭 엔진 — 편지·도감을 씀
🧩 모듈 (Module)
엔진이 호출해 구조화 데이터를 돌려주는 부품. 사용자 향 산출 아님·순수 함수 지향.
선호 계량화 · 푸드체이닝 · 영양거울 · 간식 · 두뇌
🗄️ 데이터 레이어
모듈들이 읽고 쓰는 공용 데이터. 판단 로직 없음.
추천 네트워크(food-graph) · 아이 현황 스토어
📌 표기 정리: 기존에 "푸드체이닝 엔진"·"선호 계량화 엔진"이라 부른 것은 공식적으로 모듈입니다(일상 대화에선 엔진이라 불러도 무방). 사용자 향 편지를 만드는 코칭 엔진만 '엔진'.
1

컴포넌트 지도

한 장
읽는 법 — 코칭 엔진이 콘텐츠 모듈들을 호출하고, 그 결과와 원천을 아이 현황 스토어가 매일 한 스냅샷으로 모은다. ②추천엔진이 ⑦조회 헬퍼로 추천 네트워크를 읽어(보라 점선) '다음 걸음 후보'를 만들고, 코칭 엔진은 네트워크를 직접 만지지 않는다. 통합 후 콘텐츠 모듈은 6개(①~⑥) + 조회 헬퍼 ⑦ — 상세는 §4.
⚙️ 처리 순서(코칭 엔진 내부): ① 컨텍스트 조립(②③④⑤⑦ 결정론 모듈이 CoachContext 채움) → ② 🧠 뇌(DeepSeek V4-Pro)=판단(=①신호선택기, 후보 중 오늘의 시나리오·축 택1 → decision) → ③ ✍️ 손(DeepSeek V4-Flash)=작문(=⑥). 뇌는 '입력 모듈'이 아니라 컨텍스트를 읽고 결정을 더하는 단계.
2

child_daily_state

이사님 지적대로 주간 계획표·일간 기록·편지를 매일 한곳에 모아야 코칭 엔진이 일관되게 읽습니다. 세 원천 + 모듈 파생을 야간에 한 스냅샷으로.

🗄️ child_daily_state (자녀 × 날짜)
선호 계량(잘먹음/잘못먹음·축별) · 영양 거울(식품군 신호) · 노출 카운트 · 푸드체이닝 후보집합+추천(모듈이 추천 네트워크로 산출) · 성숙도/콜드스타트/위험신호 플래그. 매일 야간 materialize(배치 1단계) → 코칭 엔진은 이 한 행만 읽는다.

3

⭐ 데이터 계약 — 코칭 엔진에 무엇을 어떻게 주나

CoachContext

코칭 엔진은 이 한 객체만 받습니다(아이 현황 스토어에서 조립). "어떤 데이터를 어떻게"의 답 — 모듈들이 채운 결정론 재료 + 푸드체이닝 다음 걸음.

// 자녀 × 하루 1객체 — 아이 현황 스토어가 조립 · asOf = 전일까지 확정 스냅샷
CoachContext = {
  meta:        { schemaVersion, asOf },                // 계약 버전 · 신선도(어제까지)
  child:       { age, allergens, maturity },
  week:        { goal, targetGroup, arcStage, lever },  // ← 주간 계획표
  today:       { meals, nutritionMirror(집+기관) },    // ← 일간 기록(전일까지)
  preference:  { liked[], disliked[], exploring[], byForm },  // ← 선호 계량화 모듈
  foodChaining:{ candidates:[ {axis,anchor,target,method,  // ← 푸드체이닝 모듈(네트워크 내부 활용)
                            exposure,sosLevel} ],          //   ★뇌가 고를 화이트리스트(목록 밖 금지)
                 recommended },                            //   결정론 디폴트(안 바꾸면 이것)
  decision:    { scenario, chainingPick, recoTrack },  // ★뇌 출력 — 후보 중 택1 · recoTrack=supply|challenge(일간 회전)
  history:     { recentLetters[], usedTargets[], feedback },  // ← 편지
  flags:       { coldStart, dataMaturity, riskScreen }  // 콜드스타트 + 위험신호 게이트
}
// 파이프라인: ① 컨텍스트 조립(결정론 모듈) → ② 뇌=decision(후보 택1) → ③ 손=작문
필드채우는 곳코칭 엔진이 쓰는 법
week주간 계획표이번 주 타깃·아크 단계에 편지 정렬(주간 일관성)
today일간 기록오늘 끼니 칭찬·영양 거울(집/기관 종합)
preference선호 계량화 모듈'잘 먹는' 단정은 여기 liked만 · disliked는 회피·형태 전환
foodChaining.candidates[]푸드체이닝 모듈뇌가 맥락으로 골라 쓸 후보집합(화이트리스트) — 목록 밖 생성 금지(괴식0)
foodChaining.recommended푸드체이닝 모듈결정론 디폴트 한 걸음 — 뇌가 안 바꾸면 이게 쓰임
decision두뇌(Sonnet) 출력오늘 시나리오 + 후보 중 택1(chainingPick). 후보 안에서만
history편지최근 문구·타깃 반복 회피 · 피드백 반영
flags선호 계량화 모듈coldStart=단정금지·온보딩(Day1) · riskScreen=위험신호 시 소아과 상담 게이트
meta아이 현황 스토어asOf=전일까지 확정(당일 입력은 다음날 반영) · schemaVersion=계약 버전
철칙 — 모듈들이 preference·foodChaining을 결정론으로 채워 계약에 담고, 코칭 엔진(손-LLM)은 그 재료로 문장만 쓴다. 코칭 엔진은 추천 네트워크도, 원천 테이블도 직접 판단하지 않는다.

🧠✍️ 모듈 출력 → 뇌·손 프롬프트 매핑

①~⑦이 채운 CoachContext가 두 프롬프트로 들어갑니다. 뇌=판단(후보 택1), 손=작문(재료→문장). 실제 라인바이라인 = 프롬프트 아키텍처.

🧠 뇌 프롬프트 (=① · DeepSeek V4-Pro)
[SYSTEM] 역할 + 13 시나리오 선택 원칙(후보 밖 금지)
[USER] ← 모듈 출력 주입:
· ④ signals[](신호등) · ③ factCards·mirror
· ②⑦ recoCandidates(승인할 추천 후보)
· ⑤ candSignals·progress·unitEvidence
· ① weeklyArc(3주 닻) · 과거편지
→ 출력(JSON): decision{scenarioId,planTarget,moveKey,useFood,approvedRecs,chainingPick}
✍️ 손 프롬프트 (=⑥ · DeepSeek V4-Flash)
[SYSTEM] P1~P10 코치 표준(편식 9원칙·안전·괴식·저염) — 고정
[USER] ← 뇌 결정 + 결정론 재료:
· ① decision(오늘 각도/타깃/무브)
· ② recoFacts·reasonPhrases(검증 음식·근거)
· ③ mirror·factCards · ④ signals
· ⑥ 만성질환 framing · 가드(연령·질식·저염)
→ 출력(JSON): {letter, oneliner}
핵심 — 뇌는 ②⑦이 만든 후보 안에서만 고르고(괴식0), 손은 ①~⑤가 깐 재료를 문장으로만 옮긴다. 두 프롬프트 모두 지어내지 않음 — 재료=결정론, 문장=LLM.

닫힌 루프 — 기록에서 다시 기록으로

flow
① 집 끼니 기록 (+수용 수준)
부모가 meal_logs 입력 — place·수용 5단계·거부 사유.
▼
② 선호 계량화 모듈
집만·노이즈 제거·축별 계량 → 잘먹음/잘못먹음/탐색(확신도).
▼
③ 푸드체이닝 모듈 (← 추천 네트워크)
확신 선호를 앵커로 사촌·곁들임·새 조리법 후보집합 + 결정론 디폴트 산출.
▼
④ 코칭 엔진 = 뇌(후보 택1) → 손(작문) → 편지·도감
뇌가 후보 중 오늘 한 걸음을 고르고(decision), 손이 작문. 콜드스타트면 온보딩.
▼
⑤ 부모 실행 → 새 끼니 → ①로
결과가 다시 기록되어 계량이 갱신 — 노출 진척·추세 학습.
아이 현황 스토어가 루프의 허브 — ②③의 출력이 매일 스토어에 적재되고 ④가 읽는다. 스토어가 비면(=선호 계량 안 됨) ③④가 허상 위에 선다. 그래서 ① 기록 + ② 계량이 1순위.
6

1만 명 배치 = 이 구조 그대로 2단계

scale

결정론 상태를 야간에 미리 계산해 스냅샷으로 두면, 편지 워커는 그걸 읽고 LLM만 돌린다 — 느린 LLM과 빠른 결정론을 분리. child_daily_state·CoachContext가 그대로 배치의 정답 형태다.

1단계 · 상태 머티리얼라이즈 결정론 · 병렬 · LLM 0
전 자녀 meal_logs(전일까지) → 선호 계량·영양거울·푸드체이닝 후보 산출 → child_daily_state upsert. 순수 함수라 대량 병렬·재실행 안전. 가장 무거운 결정론 계산을 LLM과 떼어 한 번에.
▼ 스냅샷 적재 후
2단계 · 편지 워커풀 LLM(뇌+손)
워커가 CoachContext 1행씩 읽어 뇌(후보 택1) → 손(작문). 상태가 이미 계산돼 있어 LLM 호출에만 집중 · 실패 시 그 자녀만 재시도(상태 재계산 불필요).
신선도 모델 — 스냅샷은 전일까지 확정 상태(asOf). 당일 입력은 다음 배치에 반영 — 현행 "당일 입력이 오늘 편지를 안 바꾼다"와 정확히 일치(미래·자기참조 차단). 덕분에 골든 CoachContext → 골든 편지 회귀 테스트가 가능하다.

CoachContext (TypeScript)

type Axis = 'chain'|'pair'|'method'|'temperature';
type SosLevel = 0|1|2|3|4;            // 거부·만짐·한입·조금·완식
type PrefState = 'liked'|'disliked'|'exploring'|'unknown';

interface ChainCandidate {
  axis: Axis; anchor: string; target: string;
  method?: string; exposure: number; sosLevel: SosLevel; reason: string;
}
interface CoachContext {
  meta:    { schemaVersion: number; asOf: string };        // asOf=YYYY-MM-DD(전일까지)
  child:   { id: string; ageBand: string; allergens: string[]; maturity: number };
  week:    { goal: string; targetGroup: string; arcStage: number; lever: string };
  today:   { meals: MealLog[]; nutritionMirror: Record<string,'green'|'yellow'|'red'> };
  preference: {
    liked: PrefItem[]; disliked: PrefItem[]; exploring: PrefItem[];
    byForm: Record<string, PrefState>;                     // "감자×조림" → state
  };
  foodChaining: { candidates: ChainCandidate[]; recommended: ChainCandidate | null };
  decision?: { scenario: string; chainingPick: string | null };  // 뇌가 채움
  history: { recentLetters: string[]; usedTargets: string[]; feedback?: 'up'|'down'|'retry' };
  flags:   { coldStart: boolean; dataMaturity: number; riskScreen: boolean };
}

child_daily_state · child_food_pref (SQL 초안)

-- 배치 1단계가 자녀×날짜로 upsert · 코칭 엔진(2단계)이 read
create table child_daily_state (
  child_id   uuid not null,
  as_of      date not null,            -- 전일까지 확정
  schema_ver int  not null default 1,
  preference jsonb not null,           -- liked/disliked/exploring/byForm
  nutrition  jsonb not null,           -- 식품군 신호
  food_chain jsonb not null,           -- {candidates[], recommended}
  flags      jsonb not null,           -- coldStart/dataMaturity/riskScreen
  exposure   jsonb,                       -- 식재료→노출수준·횟수
  updated_at timestamptz default now(),
  primary key (child_id, as_of)
);
-- 선호 롤업 원장(다축) — 1단계가 meal_logs(집만)에서 집계
create table child_food_pref (
  child_id uuid not null,
  axis     text not null,             -- 'ingredient'|'dish'|'form'
  key      text not null,             -- '당근' | '당근볶음' | '당근×볶음'
  served int, accepted int, rejected int, known int,
  pref_score real, confidence real,
  state text,                            -- liked/disliked/exploring/unknown
  trend text,                            -- rising/flat/falling(food jag)
  first_seen date, last_seen date, exposure int,
  inst_seen int,                         -- 기관 노출(영양용·선호 제외)
  updated_at timestamptz default now(),
  primary key (child_id, axis, key)
);
초안 — 구현 시 확정. meal_logs에 수용 수준(0~4)·구조화 거부 사유 컬럼 추가가 선행(선호 계량화 모듈 문서 §9 포착). ⭐도감 식재료별 경험 횟수 = child_food_pref(axis='ingredient')의 exposure(+last_seen·수용수준) — 별도 테이블 없이 식재료축 행으로 관리하고 meal_logs가 이벤트 원장. 폼별(감자×볶음) 경험도 axis='form' 행에 같이 쌓인다.
