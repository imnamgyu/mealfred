1. 진단 — 스캐폴드는 있으나 하네스는 아니었다

편지 생성은 이미 다단 파이프라인이다: 신호추출 → 시나리오 라우터(13종) → 프롬프트 조립 → 생성(haiku) → 안전가드 재생성 → 어휘 유사도 재생성 → 퇴고. 느슨한 의미의 스캐폴드는 맞다. 그러나 다양성을 보장하는 관절이 약했다 — 9개 에이전트가 6개 가설을 코드에 대해 반증 시도한 결과:

근본원인내용판정

재생성 비탈출임계 초과해도 같은 계획 재호출(무브 텍스트만 변경) → 근접 중복 발행 가능지지 0.95

가장 근본: source_hash = 정렬식재료+거부+reds+notes에 날짜·무브·시나리오가 빠짐. 편식 식단은 며칠씩 비슷한 게 정상이라 해시가 자주 일치 → 어제 편지를 글자 그대로 재사용하며 모든 다양성 장치를 실행조차 안 함.

2. 해법 — 계획=코드, 작문=LLM (하네스화)

핵심 전환: '무엇을 다룰지(계획)'를 LLM 손에서 빼내 코드가 결정론적으로 확정하고, LLM은 그 계획을 따뜻하게 풀어 쓰기만 한다. 계획은 구조화된 객체로 저장돼 다음날 자기 자신을 검증한다.

BEFORE · 프롬프트 적층 스캐폴드

  • 타깃 = LLM이 부족 목록에서 자유 선택
  • 검증 = 정규식 + 문자 3-gram (표면)
  • 상태 = 과거 편지 텍스트를 다시 먹여 "겹치지 마라" 호소
  • 재생성 = 같은 입력 재호출, 덜 나쁜 것 채택
  • source_hash 동결 = 다양성 장치 우회

AFTER · plan 엔진 (닫힌 루프)

  • 타깃 = 코드가 최근 안 쓴 결핍부터 결정론 회전
  • 검증 = 시그니처 frame|target|move 중복회피(생성 전)
  • 상태 = context.plan 구조화 원장(다음날 읽음)
  • 재생성 = 포화 시 정체기로 에스컬레이트
  • source_hash에 today 포함 → 매일 새 계획
